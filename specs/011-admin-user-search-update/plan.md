# Implementation Plan: Admin User Search Update

**Branch**: `[011-admin-user-search-update]` | **Date**: 2026-07-01 | **Spec**: [spec.md](file:///d:/AI/%E9%96%8B%E7%99%BA/master/PaymentRequestWorkflowManagement/specs/011-admin-user-search-update/spec.md)

**Input**: Feature specification from `/specs/011-admin-user-search-update/spec.md`

## Summary

Replace the generic keyword search box in the Admin User Management screen with two distinct fields: "Employee Number" (with an "EMP-" visual prefix) and "Employee Name". The search logic must support AND filtering when both are populated and respect a 300ms debounce.

## User Review Required

Does the proposed data model and API change align with your expectations? See `data-model.md` and `contracts/api.md`.

## Proposed Changes

### Database / API Layer (src/modules/admin/)
- No DB schema changes needed. The `User` entity remains unchanged.
- Update `admin.controller.ts` to accept `employeeNumber` and `employeeName` query parameters instead of `keyword`.
- Update `admin.service.ts` to filter using TypeORM's `Like()` or exact match (`employeeNumber`) with an AND condition if both are provided. 

### Frontend Layer (frontend/src/pages/admin/)
- Update the User Management search filter bar to use two distinct inputs.
- Add an input component with a non-editable `EMP-` prefix for the Employee Number field.
- Ensure inputs have a 300ms debounce before making API calls.
- Update API service calls in the frontend to pass `employeeNumber` and `employeeName`.
