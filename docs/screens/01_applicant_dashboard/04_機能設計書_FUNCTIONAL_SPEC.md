# 機能設計書 (Functional Specification) - 申請者ダッシュボード

**対象画面:** 申請者ダッシュボード (Applicant Dashboard)  
**バージョン:** 1.0  
**作成日:** 2026-06-12  
**ステータス:** 承認済み  

---

## 1. 機能概要 (Functional Overview)
申請者（Applicant）ロールのユーザーが、支払申請書の新規起票、下書き（Draft）の編集・保存・削除、マネージャーへの提出、および差し戻しされた申請の修正・再提出を行うためのポータル画面である。

---

## 2. ユースケースと業務フロー (Usecases & Workflow)

### 2.1 主なユースケース
1. **申請一覧の表示:** 自身が起票したすべての支払申請をステータス順・日付順で表示する。
2. **支払申請の新規作成（下書き保存）:** 支払申請に必要なヘッダー項目および最大15行の支払明細を入力し、下書きとしてデータベースに保存する。
3. **下書き申請の論理削除:** 起票中（`DRAFT`状態）の申請に限り、論理削除（アーカイブ）する。
4. **マネージャーへの提出:** 領収書ファイルが正しくアップロードされていることを検証した上で、指定したマネージャー宛に申請を提出する（ステータスが `SUBMITTED_MANAGER` に遷移）。
5. **最終承認者への送信:** マネージャーの確認が完了（`MANAGER_VERIFIED`）した申請を、最終承認者へ送信する（ステータスが `SUBMITTED_APPROVER` に遷移）。
6. **差し戻し案件の修正再提出:** マネージャーまたは最終承認者から差し戻された申請（コメントを確認）を修正し、マネージャーへ再提出する。

### 2.2 業務フロー（申請者視点）
```
[起票開始] ──► [下書き保存] ──► [領収書アップロード] 
                      │
                 (提出アクション)
                      │
                      ▼
            (領収書必須チェック)
                      ├─(NG)─► エラー表示・提出遮断
                      └─(OK)─► [マネージャー確認待ち] へ遷移
```

---

## 3. ビジネスルール (Business Rules)
1. **自己起票案件のみのアクセス制限:** 申請者は自身が起票した支払申請のみ閲覧・操作可能。他人の申請への直接アクセス（URL直打ち等）はAPIレベルで遮断する。
2. **編集可能ステータスの制限:** 編集（更新）および再提出が可能なのは、ステータスが `DRAFT`、`REJECTED_MANAGER`、`REJECTED_APPROVER` のいずれかである場合に限る。
3. **下書き削除権限:** 論理削除が可能なのは `DRAFT` 状態のみ。提出後の削除は一切禁止する。
4. **領収書ファイルの強制ルール:** 領収書の有無で「有」を選択した場合は、マネージャーへ提出する際に、アップロードファイルが1枚以上登録されていることをフロントエンド・バックエンド双方で検証する。

---

## 4. 権限制御 (Permission Control)
- **要求ロール:** `APPLICANT`
- **認証方式:** JWT Bearer トークン検証
- **認可ガード:** `RolesGuard` および `OwnershipGuard` によるアクセス制御
