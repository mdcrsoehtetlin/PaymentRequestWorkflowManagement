# 機能設計書 (Functional Specification) - 最終承認者ダッシュボード

**対象画面:** 最終承認者ダッシュボード (Approver Dashboard)  
**バージョン:** 1.0  
**作成日:** 2026-06-12  
**ステータス:** 承認済み  

---

## 1. 機能概要 (Functional Overview)
最終承認者（Final Approver）ロールのユーザーが、最終承認待ち状態（`Submitted to Approver` または `Approver Reviewing`）の支払申請案件を一覧で確認し、最終承認（Approve）または却下差し戻し（Reject）の意思決定を行うための画面である。

---

## 2. ユースケースと業務フロー (Usecases & Workflow)

### 2.1 主なユースケース
1. **最終承認待ち一覧の表示:** 自身が承認すべき最終承認待ち状態の申請を一覧表示する。
2. **申請詳細・監査証跡のレビュー:** 支払申請書情報、明細データ、領収書に加え、これまでのマネージャー確認日時やコメントを含む「承認履歴ログ（監査証跡）」を時系列で閲覧する。
3. **最終承認（Approved）処理:** 申請内容およびマネージャーの確認結果に基づき、最終承認を実行する。ステータスは `APPROVED` に遷移し、経理部門の支払キューに回送される。
4. **最終承認却下（差し戻し）処理:** 不備を検知した場合、却下コメントを入力し差し戻す。ステータスは `REJECTED_APPROVER` となり、申請者に差し戻される。

### 2.2 業務フロー（最終承認）
```
[詳細画面を開く] ──► (自動遷移: APPROVER_REVIEWING)
                            │
                      (履歴・内容精査)
                            │
               ┌────────────┴────────────┐
               ▼                         ▼
         【最終承認アクション】     【却下アクション】
               │                         │
         (経理キューへ回送)        (差し戻しコメント必須)
               │                         │
               ▼                         ▼
          [Approved]            [Rejected by Approver]
                                         │
                                   (再起票時は最初から)
                                         │
                                         ▼
                               [Submitted to Manager]
```

---

## 3. ビジネスルール (Business Rules)
1. **閲覧時ステータス自動更新:** `Submitted to Approver` の申請を最終承認者が詳細画面で開いた際、自動的にステータスを `Approver Reviewing` に更新し、ダブルアクセスや確認重複を防ぐ。
2. **却下コメントの強制:** 最終承認者が却下を行う場合、10文字以上の却下理由入力を必須とする。
3. **ワークフロー完全リセットルール:** 最終承認者が却下した案件を申請者が再提出する場合、承認チェーンはリセットされ、再度マネージャーの一次確認からやり直す必要がある。マネージャー確認を経ずに直接最終承認者に再提出することは不可とする。

---

## 4. 権限制御 (Permission Control)
- **要求ロール:** `APPROVER`
- **認証方式:** JWT Bearer トークン検証
- **認可ガード:** `RolesGuard` による制御。
