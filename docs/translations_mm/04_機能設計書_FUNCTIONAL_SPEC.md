## 1. လုပ်ဆောင်ချက် အကျဉ်းချုပ်

### 1.1 ရည်ရွယ်ချက်နှင့် နယ်ပယ်

လျှောက်ထားသူ၏ ဒက်ရှ်ဘုတ်သည် ငွေပေးချေမှု တောင်းခံလွှာ လုပ်ငန်းစဉ် စီမံခန့်ခွဲမှု စနစ်အတွင်း `APPLICANT` အခန်းကဏ္ဍ သတ်မှတ်ခံရသော အသုံးပြုသူများအတွက် အဓိက လည်ပတ်မှု ပေါ်တယ် (portal) ဖြစ်သည်။ ဤမျက်နှာပြင်သည် လျှောက်ထားသူတစ်ဦးအတွက် ၎င်းတို့ကိုယ်တိုင်၏ ငွေပေးချေမှု တောင်းခံလွှာများ၏ ဘဝစက်ဝန်း တစ်ခုလုံး — ကနဦး မူကြမ်းရေးဆွဲခြင်းမှ အတည်ပြုချက် ကွင်းဆက်သို့ နောက်ဆုံး တင်ပြခြင်းအထိ — စီမံခန့်ခွဲရန် လိုအပ်သော စွမ်းရည်အပြည့်အစုံကို ထောက်ပံ့ပေးသည်။

ဤစနစ်ခွဲသည် လုပ်ငန်းခွင် လုပ်ငန်းစဉ်အတွင်း ငွေပေးချေမှု တောင်းခံလွှာများအားလုံး စတင်ရာ ဝင်ပေါက်ဖြစ်သည်။ စနစ်တကျ အတည်ပြုစစ်ဆေးပြီးသော၊ ပြေစာတွဲထားသော၊ နှင့် စီးပွားရေး-စည်းမျဉ်း-ကိုက်ညီသော ငွေပေးချေမှု တောင်းခံလွှာများကိုသာ အောက်ပိုင်း စစ်ဆေးခြင်းနှင့် အတည်ပြုခြင်း လုပ်ငန်းစဉ်အတွင်းသို့ ဝင်ရောက်စေရန် သေချာစေရေးအတွက် ၎င်းက တာဝန်ယူသည်။

### 1.2 လုပ်ဆောင်ချက်ဆိုင်ရာ တာဝန်များ

လျှောက်ထားသူ၏ ဒက်ရှ်ဘုတ်သည် အောက်ပါ အဓိက လုပ်ဆောင်ချက်ဆိုင်ရာ နယ်ပယ်များအတွက် တာဝန်ရှိသည်-

1. **ငွေပေးချေမှု တောင်းခံလွှာ စာရင်း စီမံခန့်ခွဲမှု** — အထောက်အထား စစ်ဆေးပြီးသော လျှောက်ထားသူမှ စတင်ခဲ့သည့် ငွေပေးချေမှု တောင်းခံလွှာများအားလုံးကို အခြေအနေ ဦးစားပေးမှုနှင့် အချိန်အစီအစဉ်အလိုက် စီစဉ်ထားသော၊ စုစည်းထားသည့်၊ အချိန်နှင့်တစ်ပြေးညီ မြင်ကွင်းကို တင်ဆက်ခြင်း။
2. **ငွေပေးချေမှု တောင်းခံလွှာ မူကြမ်းရေးဆွဲခြင်း** — ခေါင်းစဉ် အချက်အလက် အပြည့်အစုံနှင့် ငွေပေးချေမှု အသေးစိတ် လိုင်းအချက်အလက် (၁၅) ခုအထိ ပါဝင်သော၊ မူကြမ်း မှတ်တမ်းများအဖြစ် သိမ်းဆည်းထားသော ငွေပေးချေမှု တောင်းခံလွှာအသစ်များ ဖန်တီးခြင်းကို လုပ်ဆောင်နိုင်စေခြင်း။
3. **မူကြမ်း ဘဝစက်ဝန်း စီမံခန့်ခွဲမှု** — ငွေပေးချေမှု တောင်းခံလွှာများသည် `DRAFT` အခြေအနေတွင် ရှိနေစဉ် ၎င်းတို့အား တည်းဖြတ်ခြင်း၊ မွမ်းမံခြင်း၊ နှင့် ယုတ္တိဗေဒဆိုင်ရာ (soft) ဖျက်ပစ်ခြင်းတို့ကို ထောက်ပံ့ပေးခြင်း။
4. **ပြေစာ ဖိုင် ပူးတွဲခြင်း** — ဖိုင်အမည်ပေးခြင်း စည်းမျဉ်းများနှင့် အရွယ်အစား ကန့်သတ်ချက်များ သတ်မှတ်ထားရှိပြီး ငွေပေးချေမှု တောင်းခံလွှာများတွင် ပြေစာ အထောက်အထားများ (PDF, PNG, JPG, JPEG) ပူးတွဲရန်အတွက် ဖိုင်တင်ပို့ခြင်း (file upload) စွမ်းရည်များကို ထောက်ပံ့ပေးခြင်း။
5. **မန်နေဂျာထံ တင်ပြခြင်း** — ငွေပေးချေမှု တောင်းခံလွှာအား မန်နေဂျာမှ စစ်ဆေးရန်အတွက် `SUBMITTED_MANAGER` အခြေအနေသို့ မပြောင်းလဲမီ မဖြစ်မနေ လိုအပ်သော အကွက်များအားလုံးနှင့် ပြေစာ ပူးတွဲမှု လိုအပ်ချက်များကို အတည်ပြုစစ်ဆေးခြင်း။
6. **နောက်ဆုံး အတည်ပြုသူထံ တင်ပြခြင်း** — မန်နေဂျာ၏ စစ်ဆေးမှု အောင်မြင်သည်နှင့် (အခြေအနေ `MANAGER_VERIFIED`)၊ အခြေအနေကို `SUBMITTED_APPROVER` သို့ ပြောင်းလဲခြင်းဖြင့် လျှောက်ထားသူမှ တောင်းခံလွှာအား နောက်ဆုံး အတည်ပြုသူထံ ပေးပို့ရန် လုပ်ဆောင်နိုင်စေခြင်း။
7. **ပယ်ချခံရခြင်းကို တုံ့ပြန်ခြင်းနှင့် ပြန်လည်တင်ပြခြင်း** — မန်နေဂျာ သို့မဟုတ် နောက်ဆုံး အတည်ပြုသူထံမှ ပယ်ချသည့် မှတ်ချက်များကို ပြသခြင်း၊ နှင့် လျှောက်ထားသူမှ ပြင်ဆင်ပြီးသော တောင်းခံလွှာများကို ပြောင်းလဲ၍ ပြန်လည်တင်ပြရန် လုပ်ဆောင်နိုင်စေခြင်း။
8. **အတည်ပြုချက် အချိန်ဇယား မြင်သာမှု** — ငွေပေးချေမှု တောင်းခံလွှာတစ်ခုစီနှင့် သက်ဆိုင်သော အတည်ပြုချက် လုပ်ဆောင်မှုများအားလုံး၊ အခြေအနေ ပြောင်းလဲမှုများ၊ နှင့် သုံးသပ်သူ၏ မှတ်ချက်များအားလုံး၏ အချိန်အစီအစဉ်အလိုက် အချိန်ဇယားကို တင်ဆက်ခြင်း။

## 2. အသုံးပြုမှုဖြစ်စဉ်များနှင့် စီးပွားရေးလုပ်ငန်း လုပ်ငန်းစဉ်

### 2.1 အသုံးပြုမှုဖြစ်စဉ် ကက်တလောက်

အောက်ပါဇယားသည် ဤလုပ်ဆောင်ချက်ဆိုင်ရာ သတ်မှတ်ချက်မှ အုပ်ချုပ်သော အသုံးပြုမှုဖြစ်စဉ်များအားလုံးကို ရေတွက်ပြထားသည်။ လိုအပ်ချက်များ သတ်မှတ်ချက်နှင့် စမ်းသပ်မှုဖြစ်စဉ် သတ်မှတ်ချက်များနှင့် ခြေရာခံနိုင်ရန်အတွက် အသုံးပြုမှုဖြစ်စဉ်တစ်ခုစီကို သီးခြား အမှတ်အသားတစ်ခု သတ်မှတ်ပေးထားသည်။

### 3.3 တည်းဖြတ်နိုင်သော အခြေအနေ အစုအဝေး သတ်မှတ်ချက်

- `REJECTED_MANAGER` (status_id: 5)
- `REJECTED_APPROVER` (status_id: 9)

### 5.1 လည်ပတ်မှု: ငွေပေးချေမှု တောင်းခံလွှာ စာရင်း ပြသခြင်း

| ဂုဏ်သတ္တိ | သတ်မှတ်ချက် |
| :--- | :--- |
| **အစပျိုးခြင်း** | လျှောက်ထားသူသည် Dashboard မျက်နှာပြင်သို့ သွားသည်၊ သို့မဟုတ် မျက်နှာပြင် refresh ဖြစ်ရပ် ဖြစ်ပေါ်သည်။ |
| **ဒေတာ အရင်းအမြစ်** | အထောက်အထား စစ်ဆေးပြီးသော အသုံးပြုသူ၏ အကြောင်းအရာ (context) နှင့်အတူ `GET /api/payment-requests/my-requests`။ |
| **မေးမြန်းမှု စစ်ထုတ်ခြင်း** | `applicant_user_id = current_user.user_id AND is_deleted = FALSE` |
| **ပုံသေ စီစဉ်မှု (Sort)** | ပထမ- `status_id` (`payment_statuses.display_order` အရ ကြီးစဉ်ငယ်လိုက် ပြသမှု အစီအစဉ်)၊ ဒုတိယ- `created_date` (ကြီးစဉ်ငယ်လိုက် - နောက်ဆုံးဖန်တီးထားသည်ကို အရင်ပြ)။ |
| **စာမျက်နှာခွဲခြားခြင်း (Pagination)** | ဆာဗာဘက်မှ စာမျက်နှာခွဲခြားခြင်း။ ပုံသေ စာမျက်နှာ အရွယ်အစား: မှတ်တမ်း ၁၀ ခု။ Query parameters: `page` (ပုံသေ: 1), `limit` (ပုံသေ: 10)။ |
| **အခြေအနေ စစ်ထုတ်ခြင်း** | ရွေးချယ်နိုင်သော။ ကလိုင်းယင့်ဘက်မှ အခြေအနေ စစ်ထုတ်ခြင်း dropdown သည် တိကျသော `status_code` တန်ဖိုးများဖြင့် စစ်ထုတ်ခွင့်ပြုသည်။ |
| **ပြသမည့် အကွက်များ** | တောင်းခံလွှာ နံပါတ် (`request_number`)၊ လျှောက်ထားသည့် ရက်စွဲ (`application_date`)၊ စုစုပေါင်း ပမာဏ (`total_amount`)၊ ငွေကြေး ကုဒ် (`currencies.currency_code` မှ ရယူသည်)၊ အခြေအနေ အမည် (`payment_statuses.status_name` မှ ရယူသည်)၊ ဖန်တီးသည့် ရက်စွဲ (`created_date`)။ |
| **အချိန်နှင့်တစ်ပြေးညီ အသစ်ပြင်ဆင်ခြင်း** | လက်ရှိ အသုံးပြုသူမှ ပိုင်ဆိုင်သော မည်သည့် တောင်းခံလွှာအတွက်မဆို WebSocket `statusUpdate` ဖြစ်ရပ် လက်ခံရရှိသည့်အခါ စာရင်းသည် အလိုအလျောက် refresh ဖြစ်သည်။ |

### 5.2 လည်ပတ်မှု: ငွေပေးချေမှု တောင်းခံလွှာအသစ် ဖန်တီးခြင်း (မူကြမ်း သိမ်းဆည်းခြင်း)

| ဂုဏ်သတ္တိ | သတ်မှတ်ချက် |
| :--- | :--- |
| **အစပျိုးခြင်း** | လျှောက်ထားသူသည် "Create New Request" (新規申請作成) ခလုတ်ကို နှိပ်သည်။ |
| **API Endpoint** | `POST /api/payment-requests` |
| **တောင်းဆိုမှု Content-Type** | `application/json` |
| **အထောက်အထား စစ်ဆေးမှု** | `Authorization: Bearer <JWT>` ခေါင်းစဉ် (header) လိုအပ်သည်။ |
| **Payload ဖွဲ့စည်းပုံ** | ခေါင်းစဉ် အကွက်များ (`application_date`, `desired_payment_date`, `currency_id`, `payment_type_id`, `payment_method_id`, `purpose`, `bank_account_info`, `request_content`, `has_receipt`, `manager_user_id`) နှင့် အသေးစိတ် လိုင်းအချက်အလက်များပါဝင်သော ထပ်ဆင့် `items` array (`line_number`, `item_date`, `description`, `amount`, `quantity`, `unit_price` တစ်ခုစီပါဝင်သည်) ပါဝင်သော JSON object။ |
| **လုပ်ဆောင်မှု အဆင့်များ** | 1. အထောက်အထား စစ်ဆေးပြီးသော အသုံးပြုသူတွင် `APPLICANT` အခန်းကဏ္ဍ ရှိကြောင်း အတည်ပြုစစ်ဆေးပါ။ 2. `PRF-YYYY-NNNNNN` ပုံစံဖြင့် ထူးခြားသော `request_number` ကို ဖန်တီးပါ။ 3. JWT payload မှ `applicant_user_id` ကို သတ်မှတ်ပါ။ 4. `status_id = 1 (DRAFT)` ကို သတ်မှတ်ပါ။ 5. လိုင်းအချက်အလက် ပမာဏများ၏ စုစုပေါင်းမှ `total_amount` ကို တွက်ချက်ပါ။ 6. `payment_requests` မှတ်တမ်းကို သိမ်းဆည်းပါ။ 7. ဆက်စပ်နေသော `payment_breakdown_items` မှတ်တမ်းများကို သိမ်းဆည်းပါ။ 8. `action_type_id` = `CREATED` ဖြင့် `approval_logs` မှတ်တမ်းကို ရေးပါ။ 9. အသစ်ဖန်တီးထားသော တောင်းခံလွှာ object နှင့်အတူ `201 Created` ကို ပြန်လည်ပေးပို့ပါ။ |
| **မူကြမ်း သိမ်းဆည်းမှု အပြုအမူ** | မူကြမ်း သိမ်းဆည်းခြင်းသည် မဖြစ်မနေ ထည့်သွင်းရမည့် အကွက်များအားလုံး၏ အတည်ပြုစစ်ဆေးမှုကို မပြဋ္ဌာန်းပါ။ အကွက်များကို တစ်စိတ်တစ်ပိုင်းသာ ဖြည့်စွက်ထားနိုင်သည်။ မတူညီသော ဝင်ရောက်မှု အကြိမ်ကြိမ် (sessions) အတွင်း ပုံစံကို တဖြည်းဖြည်း ဖြည့်စွက်နိုင်ရန် အတည်ပြုစစ်ဆေးမှုကို လျှော့ပေါ့ထားသည်။ |
| **တောင်းခံလွှာ နံပါတ် ဖန်တီးခြင်း** | ပုံစံ: `PRF-{YYYY}-{NNNNNN}`၊ ယင်းတွင် `YYYY` သည် ဂဏန်းလေးလုံးပါ နှစ်ဖြစ်ပြီး `NNNNNN` သည် သုညဖြင့် ဖြည့်ထားသော အစဉ်လိုက် နံပါတ်ဖြစ်သည်။ ထူးခြားမှုကို `uq_payment_requests_number` ဒေတာဘေ့စ် ကန့်သတ်ချက်ဖြင့် ပြဋ္ဌာန်းထားသည်။ ပုံစံကို `chk_payment_requests_number_format` CHECK ကန့်သတ်ချက်ဖြင့် အတည်ပြုစစ်ဆေးသည်: `^PRF-[0-9]{4}-[0-9]{3,6}$`။ |

### 5.3 လည်ပတ်မှု: ရှိပြီးသား ငွေပေးချေမှု တောင်းခံလွှာကို တည်းဖြတ်ခြင်း

| ဂုဏ်သတ္တိ | သတ်မှတ်ချက် |
| :--- | :--- |
| **အစပျိုးခြင်း** | လျှောက်ထားသူသည် စာရင်းထဲမှ တည်းဖြတ်နိုင်သော အခြေအနေရှိ တောင်းခံလွှာကို ရွေးချယ်သည်။ |
| **အစောင့် စည်းမျဉ်း** | တောင်းခံလွှာ `status_id` ∈ {1, 5, 9} (တည်းဖြတ်နိုင်သော အခြေအနေများ)။ ပိုင်ဆိုင်မှုကို အတည်ပြုပြီးသည် (`applicant_user_id = current_user.user_id`)။ |
| **API Endpoint** | `PATCH /api/payment-requests/:id` |
| **လုပ်ဆောင်မှု အဆင့်များ** | 1. ပိုင်ဆိုင်မှုနှင့် တည်းဖြတ်နိုင်သော အခြေအနေကို အတည်ပြုစစ်ဆေးပါ။ 2. အကွက်-အဆင့် ပြင်ဆင်ချက်များကို `payment_requests` မှတ်တမ်းသို့ အသုံးပြုပါ။ 3. `payment_breakdown_items` ကို ညှိနှိုင်းပါ (အသစ်များကို ထည့်သွင်းပါ၊ ရှိပြီးသားများကို အသစ်ပြင်ဆင်ပါ၊ ဖယ်ရှားထားသည်များကို ဖျက်ပါ)။ 4. `total_amount` ကို ပြန်လည်တွက်ချက်ပါ။ 5. `modified_date` ကို အသစ်ပြင်ဆင်ပါ။ 6. `action_type_id` = `EDITED` ဖြင့် `approval_logs` မှတ်တမ်းကို ရေးပါ။ 7. အသစ်ပြင်ဆင်ထားသော တောင်းခံလွှာ object နှင့်အတူ `200 OK` ကို ပြန်လည်ပေးပို့ပါ။ |
| **တစ်ပြိုင်နက်တည်း လုပ်ဆောင်မှု ထိန်းချုပ်ခြင်း (Concurrency Control)** | `modified_date` နှိုင်းယှဉ်ချက်ကို အသုံးပြု၍ အကောင်းမြင်သော တစ်ပြိုင်နက်တည်း လုပ်ဆောင်မှု ထိန်းချုပ်ခြင်း (Optimistic concurrency) ကို အသုံးပြုသည်။ ဆာဗာဘက်မှ `modified_date` သည် ကလိုင်းယင့်၏ နောက်ဆုံး-သိထားသော တန်ဖိုးနှင့် ကွဲလွဲနေပါက၊ ပြင်ဆင်ချက်ကို HTTP `409 Conflict` ဖြင့် ပယ်ချပြီး၊ အသုံးပြုသူအား စာမျက်နှာကို refresh လုပ်၍ ပြန်လည်ကြိုးစားရန် ညွှန်ကြားသည့် မက်ဆေ့ဂျ်တစ်စောင် ပြသသည်။ |

### 5.4 လည်ပတ်မှု: မူကြမ်းကို Soft Delete လုပ်ခြင်း

| ဂုဏ်သတ္တိ | သတ်မှတ်ချက် |
| :--- | :--- |
| **အစပျိုးခြင်း** | လျှောက်ထားသူသည် `DRAFT` အခြေအနေရှိ တောင်းခံလွှာပေါ်ရှိ "Delete Draft" (下書きを削除) ခလုတ်ကို နှိပ်သည်။ |
| **အစောင့် စည်းမျဉ်း** | တောင်းခံလွှာ `status_id = 1 (DRAFT)` သီးသန့်ဖြစ်ရမည်။ ပိုင်ဆိုင်မှုကို အတည်ပြုပြီးသည်။ |
| **API Endpoint** | `DELETE /api/payment-requests/:id` |
| **အသုံးပြုသူ အတည်ပြုချက်** | API ကို မခေါ်ယူမီ အတည်ပြုချက် ဒိုင်ယာလော့ (modal confirmation dialog) ကို ပြသသည်။ ဒိုင်ယာလော့ စာသား: "ဤလုပ်ဆောင်ချက်သည် ဤမူကြမ်းကို အပြီးတိုင် မှတ်တမ်းအဖြစ် သိမ်းဆည်းမည်ဖြစ်သည်။ မူကြမ်းကို သင်၏ အသက်ဝင်နေသော စာရင်းမှ ဖယ်ရှားမည်ဖြစ်ပြီး ပြန်လည်ရယူ၍ မရနိုင်ပါ။ ဆက်လက် လုပ်ဆောင်လိုပါသလား?" ခလုတ်များ: "Confirm Delete" (အဓိက လုပ်ဆောင်ချက်), "Cancel" (ပယ်ဖျက်ရန်)။ |
| **လုပ်ဆောင်မှု အဆင့်များ** | 1. ပိုင်ဆိုင်မှုနှင့် `DRAFT` အခြေအနေကို အတည်ပြုစစ်ဆေးပါ။ 2. `is_deleted = TRUE` ဟု သတ်မှတ်ပါ။ 3. `modified_date` ကို အသစ်ပြင်ဆင်ပါ။ 4. အောင်မြင်ကြောင်း အတည်ပြုချက်နှင့်အတူ `200 OK` ကို ပြန်လည်ပေးပို့ပါ။ 5. အရှေ့ပိုင်း (frontend) သည် ပြသထားသော စာရင်းမှ ပစ္စည်းကို ဖယ်ရှားသည်။ |
| **ပယ်ချသည့် အပြုအမူ** | တောင်းခံလွှာသည် `DRAFT` အခြေအနေတွင် မရှိပါက၊ အမှားပေးပို့ချက်နှင့်အတူ HTTP `403 Forbidden` ကို ပြန်လည်ပေးပို့ပါ: `{ "error": "FORBIDDEN", "message": "Deletion is only permitted for requests in DRAFT status." }` |

### 5.5 လည်ပတ်မှု: ပြေစာ ဖိုင်ကို Upload လုပ်ခြင်း

| ဂုဏ်သတ္တိ | သတ်မှတ်ချက် |
| :--- | :--- |
| **အစပျိုးခြင်း** | ငွေပေးချေမှု တောင်းခံလွှာ ပုံစံပေါ်ရှိ ပြေစာ upload interface မှတစ်ဆင့် လျှောက်ထားသူသည် ဖိုင်များကို ရွေးချယ်သည်။ |
| **API Endpoint** | `POST /api/payment-requests/:id/receipts` |
| **တောင်းဆိုမှု Content-Type** | `multipart/form-data` |
| **Payload** | ဘိုင်နရီ ဖိုင် ဒေတာ ပါဝင်သော `file` အကွက်။ |
| **လုပ်ဆောင်မှု အဆင့်များ** | 1. ပိုင်ဆိုင်မှုကို အတည်ပြုစစ်ဆေးပါ။ 2. MIME အမျိုးအစားကို ခွင့်ပြုထားသော အမျိုးအစားများ (PDF, PNG, JPG, JPEG) ဖြင့် အတည်ပြုစစ်ဆေးပါ။ 3. ဖိုင် အရွယ်အစား ≤ 10 MB ကို အတည်ပြုစစ်ဆေးပါ။ 4. စုစုပေါင်း ပူးတွဲမှု အရွယ်အစား ≤ 50 MB ကို အတည်ပြုစစ်ဆေးပါ။ 5. အသေးစိတ် အကြောင်းအရာ ဖိုင်အမည်ပေးခြင်း စည်းမျဉ်းကို လိုက်နာ၍ `stored_file_name` ကို ဖန်တီးပါ။ 6. ဖိုင်ကို `/uploads/{payment_request_id}/{UUID}_{stored_file_name}` သို့ သိမ်းဆည်းပါ။ 7. ဖိုင် မက်တာဒေတာ (metadata) ဖြင့် `receipt_files` မှတ်တမ်းကို ဖန်တီးပါ။ 8. ပြေစာ ဖိုင် မက်တာဒေတာ object နှင့်အတူ `201 Created` ကို ပြန်လည်ပေးပို့ပါ။ |
| **တုံ့ပြန်မှု Payload** | `{ "receipt_file_id": <int>, "original_file_name": "<string>", "stored_file_name": "<string>", "file_storage_path": "<string>", "file_size": <bigint>, "mime_type": "<string>" }` |

### 5.6 လည်ပတ်မှု: မန်နေဂျာထံ တင်ပြခြင်း

| ဂုဏ်သတ္တိ | သတ်မှတ်ချက် |
| :--- | :--- |
| **အစပျိုးခြင်း** | လျှောက်ထားသူသည် "Submit to Manager" (マネージャーへ提出) ခလုတ်ကို နှိပ်သည်။ |
| **API Endpoint** | `POST /api/payment-requests/:id/submit-manager` |
| **တင်ပြခြင်းမပြုမီ အတည်ပြုစစ်ဆေးမှု** | မဖြစ်မနေ ထည့်သွင်းရမည့် အကွက်များအားလုံး၏ အတည်ပြုစစ်ဆေးမှုကို အပြည့်အဝ ပြဋ္ဌာန်းသည် (အပိုင်း ၆ ကို ကြည့်ပါ)။ ပြေစာ ဖိုင် ပူးတွဲမှု အတည်ပြုစစ်ဆေးမှုကို BR-APP-009 အရ ပြဋ္ဌာန်းသည်။ |
| **လုပ်ဆောင်မှု အဆင့်များ** | 1. ပိုင်ဆိုင်မှုကို အတည်ပြုစစ်ဆေးပါ။ 2. လက်ရှိ အခြေအနေ ∈ {`DRAFT`, `REJECTED_MANAGER`, `REJECTED_APPROVER`} ကို အတည်ပြုစစ်ဆေးပါ။ 3. အကွက် အတည်ပြုစစ်ဆေးမှု အစုံလိုက် အပြည့်အစုံကို အလုပ်လုပ်ပါ။ 4. `has_receipt = TRUE` ဖြစ်ပါက ပြေစာ ပူးတွဲမှုကို စစ်ဆေးအတည်ပြုပါ။ 5. ဒေတာဘေ့စ် လွှဲပြောင်းမှု (transaction) ကို စတင်ပါ။ 6. `status_id = 2 (SUBMITTED_MANAGER)` ကို အသစ်ပြင်ဆင်ပါ။ 7. `current_assigned_to_user_id = manager_user_id` ဟု သတ်မှတ်ပါ။ 8. `submitted_to_manager_date = NOW()` ကို မှတ်တမ်းတင်ပါ။ 9. `action_type_id` = `SUBMITTED`, `previous_status_id` = (ယခင် အခြေအနေ), `new_status_id = 2` ဖြင့် `approval_logs` မှတ်တမ်းကို ရေးပါ။ 10. လွှဲပြောင်းမှုကို အတည်ပြု (Commit) ပါ။ 11. WebSocket `statusUpdate` ဖြစ်ရပ်ကို မန်နေဂျာ အခန်းကဏ္ဍ အခန်းနှင့် တစ်ဦးချင်း မန်နေဂျာ၏ အသုံးပြုသူ အခန်း (`user:{manager_user_id}`) သို့ ပေးပို့ပါ။ 12. အသစ်ပြင်ဆင်ထားသော တောင်းခံလွှာ object နှင့်အတူ `200 OK` ကို ပြန်လည်ပေးပို့ပါ။ |

### 5.7 လည်ပတ်မှု: နောက်ဆုံး အတည်ပြုသူထံ တင်ပြခြင်း

| ဂုဏ်သတ္တိ | သတ်မှတ်ချက် |
| :--- | :--- |
| **အစပျိုးခြင်း** | လျှောက်ထားသူသည် "Submit to Final Approver" (最終承認者へ送信) ခလုတ်ကို နှိပ်သည်။ |
| **အစောင့် စည်းမျဉ်း** | တောင်းခံလွှာ `status_id = 4 (MANAGER_VERIFIED)` သီးသန့်ဖြစ်ရမည်။ |
| **API Endpoint** | `POST /api/payment-requests/:id/submit-approver` |
| **လုပ်ဆောင်မှု အဆင့်များ** | 1. ပိုင်ဆိုင်မှုကို အတည်ပြုစစ်ဆေးပါ။ 2. လက်ရှိ အခြေအနေသည် `MANAGER_VERIFIED` (4) အတိအကျဖြစ်ကြောင်း အတည်ပြုစစ်ဆေးပါ။ 3. ဒေတာဘေ့စ် လွှဲပြောင်းမှု (transaction) ကို စတင်ပါ။ 4. `status_id = 6 (SUBMITTED_APPROVER)` ကို အသစ်ပြင်ဆင်ပါ။ 5. `submitted_to_approver_date = NOW()` ကို မှတ်တမ်းတင်ပါ။ 6. `action_type_id` = `SUBMITTED`, `previous_status_id = 4`, `new_status_id = 6` ဖြင့် `approval_logs` မှတ်တမ်းကို ရေးပါ။ 7. လွှဲပြောင်းမှုကို အတည်ပြု (Commit) ပါ။ 8. WebSocket `statusUpdate` ဖြစ်ရပ်ကို နောက်ဆုံး အတည်ပြုသူ အခန်းကဏ္ဍ အခန်းသို့ ပေးပို့ပါ။ 9. အသစ်ပြင်ဆင်ထားသော တောင်းခံလွှာ object နှင့်အတူ `200 OK` ကို ပြန်လည်ပေးပို့ပါ။ |
| **ခလုတ် မြင်သာမှု** | "Submit to Final Approver" ခလုတ်ကို တောင်းခံလွှာ၏ လက်ရှိ အခြေအနေသည် `MANAGER_VERIFIED` (4) ဖြစ်နေချိန်တွင်သာ သီးသန့် ပြသသည်။ အခြား အခြေအနေများအားလုံးအတွက်၊ ဤခလုတ်ကို ဖျောက်ထားသည်။ |

## 6. ထည့်သွင်းမှု အတည်ပြုစစ်ဆေးခြင်း စည်းမျဉ်းများ

### 6.1 မူကြမ်း သိမ်းဆည်းခြင်း အတည်ပြုစစ်ဆေးမှု (လျှော့ပေါ့ထားသော မုဒ်)

မူကြမ်း သိမ်းဆည်းသည့် လုပ်ဆောင်မှုများအတွင်း၊ စနစ်သည် လျှော့ပေါ့ထားသော အတည်ပြုစစ်ဆေးမှုကို အသုံးပြုသည်။ မူကြမ်း သိမ်းဆည်းစဉ်အတွင်း အောက်ပါ စည်းမျဉ်းများကို ပြဋ္ဌာန်းသည်-

| အကွက် | အတည်ပြုစစ်ဆေးခြင်း စည်းမျဉ်း | အမှားအယွင်း အခြေအနေ |
| :--- | :--- | :--- |
| `application_date` | ထည့်သွင်းထားပါက၊ မှန်ကန်သော ရက်စွဲ ≤ ယနေ့ ဖြစ်ရမည်။ | အနာဂတ် ရက်စွဲကို ထည့်သွင်းထားသည်။ |
| `desired_payment_date` | ထည့်သွင်းထားပါက၊ မှန်ကန်သော ရက်စွဲ ≥ ယနေ့ ဖြစ်ရမည်။ | အတိတ် ရက်စွဲကို ထည့်သွင်းထားသည်။ |
| `amount` (လိုင်းအချက်အလက်များ) | ထည့်သွင်းထားပါက၊ ဂဏန်းဖြစ်ရမည်ဖြစ်ပြီး > 0 ဖြစ်ရမည်။ | ဂဏန်းမဟုတ်သော သို့မဟုတ် ≤ 0 တန်ဖိုး။ |
| `line_number` | 1–15 အတိုင်းအတာအတွင်း ဖြစ်ရမည်။ | အတိုင်းအတာ ပြင်ပ တန်ဖိုး။ |

အခြား အကွက်များအားလုံးကို မူကြမ်း သိမ်းဆည်းစဉ်အတွင်း အတည်ပြုစစ်ဆေးမှု အမှားအယွင်းများ မဖြစ်စေဘဲ အလွတ်ထားခဲ့နိုင်သည်။

### 6.2 တင်ပြခြင်း အတည်ပြုစစ်ဆေးမှု (တင်းကျပ်သော မုဒ်)

မန်နေဂျာထံ တင်ပြသောအခါ (UC-APP-007)၊ စနစ်သည် အတည်ပြုစစ်ဆေးမှု အစုံလိုက် အပြည့်အစုံကို ပြဋ္ဌာန်းသည်။ အခြေအနေ ပြောင်းလဲခြင်းကို ခွင့်မပြုမီ အောက်ဖော်ပြပါ အကွက်တိုင်းသည် အတည်ပြုစစ်ဆေးမှု စည်းမျဉ်းများကို အောင်မြင်ရမည်။

| အပိုင်း | အကွက် (ရုပ်ပိုင်းဆိုင်ရာ အမည်) | ပြသမည့် အမည် | အတည်ပြုစစ်ဆေးခြင်း စည်းမျဉ်း | အမှားပေးပို့ချက် မက်ဆေ့ဂျ် |

### 6.3 အတည်ပြုစစ်ဆေးမှု ပြဋ္ဌာန်းသည့် အလွှာများ

နက်ရှိုင်းစွာ ကာကွယ်မှု (defense-in-depth) သေချာစေရန် ဤအပိုင်းတွင် သတ်မှတ်ထားသော အတည်ပြုစစ်ဆေးခြင်း စည်းမျဉ်းများအားလုံးကို လွတ်လပ်သော အလွှာနှစ်ခုတွင် ပြဋ္ဌာန်းထားသည်-

| အလွှာ | နည်းပညာ | တာဝန် |
| :--- | :--- | :--- |
| **အရှေ့ပိုင်း (ကလိုင်းယင့်)** | အချိန်နှင့်တစ်ပြေးညီ တုံ့ပြန်ချက်ပါရှိသော React ပုံစံ အတည်ပြုစစ်ဆေးမှု | ထည့်သွင်းမှု အမှားအယွင်းများအပေါ် ချက်ချင်း အသုံးပြုသူထံ တုံ့ပြန်ချက် ပေးသည်။ မမှန်ကန်သော ပုံစံများကို တင်ပြခြင်းမှ ကာကွယ်သည်။ မလိုအပ်သော API ခေါ်ယူမှုများကို လျှော့ချပေးသည်။ |
| **အနောက်ပိုင်း (ဆာဗာ)** | NestJS Service Layer + class-validator DTOs | အာဏာပိုင် အတည်ပြုစစ်ဆေးမှု။ မမှန်ကန်သော payload များကို HTTP `400 Bad Request` နှင့် ဖွဲ့စည်းထားသော အမှား တုံ့ပြန်မှုဖြင့် ပယ်ချသည်။ ကလိုင်းယင့်ဘက်မှ အတည်ပြုစစ်ဆေးမှုကို ကျော်ဖြတ်ခြင်းမှ ကာကွယ်သည်။ |

## 7. အမှားအယွင်း ကိုင်တွယ်ဖြေရှင်းခြင်း သတ်မှတ်ချက်

### 7.1 အမှား တုံ့ပြန်မှု ဖွဲ့စည်းပုံ

အနောက်ပိုင်းမှ API အမှား တုံ့ပြန်မှုများအားလုံးသည် အောက်ပါ JSON ဖွဲ့စည်းပုံနှင့် ကိုက်ညီရမည်-

```json
{
  "statusCode": 400,
  "error": "BAD_REQUEST",
  "message": "Validation failed for the submitted payment request.",
  "details": [
    {
      "field": "desired_payment_date",
      "constraint": "isDateAfterToday",
      "message": "Desired Payment Date must be today or a future date."
    }
  ],
  "timestamp": "2026-06-12T08:45:00.000Z",
  "path": "/api/payment-requests"
}
```

### 7.2 အမှား အမျိုးအစားခွဲခြားခြင်း ဇယား

| HTTP အခြေအနေ | အမှား ကုဒ် | အခြေအနေ | အသုံးပြုသူ-ရင်ဆိုင်ရမည့် အပြုအမူ |
| :---: | :--- | :--- | :--- |
| `400` | `BAD_REQUEST` | တင်ပြထားသော ဒေတာတွင် အတည်ပြုစစ်ဆေးမှု ကျရှုံးခြင်း။ အကွက်တစ်ခု သို့မဟုတ် တစ်ခုထက်ပိုသော အကွက်များသည် အတည်ပြုစစ်ဆေးခြင်း စည်းမျဉ်းများကို ကျရှုံးခဲ့သည်။ | မမှန်ကန်သော အကွက်တစ်ခုစီ၏ ဘေးတွင် လိုင်းတွင်း အကွက်-အဆင့် အမှား မက်ဆေ့ဂျ်များကို ပြသပါ။ ပုံစံ၏ အပေါ်ဆုံးတွင် အကျဉ်းချုပ် အမှား ဘန်နာ (banner) တစ်ခုကို ပြသပါ။ |
| `401` | `UNAUTHORIZED` | JWT တိုကင် ပျောက်ဆုံးနေသည်၊ သက်တမ်းကုန်သွားသည်၊ သို့မဟုတ် မမှန်ကန်ပါ။ | အသုံးပြုသူအား ဝင်ရောက်ရန် (login) မျက်နှာပြင်သို့ ပြန်လည်လမ်းညွှန်ပါ။ ဝင်ရောက်မှု အကြိမ် (session) သက်တမ်းကုန်ဆုံးကြောင်း မက်ဆေ့ဂျ်ကို ပြသပါ။ |
| `403` | `FORBIDDEN` | အသုံးပြုသူသည် ၎င်းတို့ မပိုင်ဆိုင်သော အရင်းအမြစ်တစ်ခုကို ဝင်ရောက်ရန် သို့မဟုတ် ပြင်ဆင်ရန် ကြိုးစားသည်၊ သို့မဟုတ် တားမြစ်ထားသော လုပ်ဆောင်ချက်တစ်ခုကို ကြိုးပမ်းသည် (ဥပမာ၊ မူကြမ်းမဟုတ်သော တောင်းခံလွှာကို ဖျက်ခြင်း)။ | ဝင်ရောက်ခွင့် ငြင်းပယ်ခံရကြောင်း မက်ဆေ့ဂျ်ကို ပြသပါ။ ပစ်မှတ် အရင်းအမြစ်နှင့်ပတ်သက်သော အသေးစိတ် အချက်အလက်များကို မဖွင့်ဟပါနှင့်။ |
| `404` | `NOT_FOUND` | သတ်မှတ်ထားသော ငွေပေးချေမှု တောင်းခံလွှာ ID မတည်ရှိပါ သို့မဟုတ် soft-delete လုပ်ခံထားရသည်။ | "The requested payment request was not found or has been archived." (တောင်းဆိုထားသော ငွေပေးချေမှု တောင်းခံလွှာကို ရှာမတွေ့ပါ သို့မဟုတ် မှတ်တမ်းအဖြစ် သိမ်းဆည်းထားသည်။) ဟု ပြသပါ။ |
| `409` | `CONFLICT` | အကောင်းမြင်သော တစ်ပြိုင်နက်တည်း လုပ်ဆောင်မှု (Optimistic concurrency) ပဋိပက္ခ။ အသုံးပြုသူ၏ နောက်ဆုံး ဖတ်ရှုမှုအပြီးတွင် မှတ်တမ်းကို အခြား ဝင်ရောက်မှု တစ်ခုမှ ပြင်ဆင်ခဲ့သည်။ | "This request has been modified since you last loaded it. Please refresh the page and try again." (ဤတောင်းခံလွှာကို သင်နောက်ဆုံးဖွင့်ပြီးကတည်းက ပြင်ဆင်ထားသည်။ ကျေးဇူးပြု၍ စာမျက်နှာကို refresh လုပ်ပြီး ပြန်လည်ကြိုးစားပါ။) ဟု ပြသပါ။ |
| `413` | `PAYLOAD_TOO_LARGE` | Upload လုပ်ထားသော ပြေစာ ဖိုင်သည် ၁၀ MB အရွယ်အစား ကန့်သတ်ချက်ထက် ကျော်လွန်နေသည်။ | "The selected file exceeds the maximum allowed size of 10 MB. Please select a smaller file." (ရွေးချယ်ထားသော ဖိုင်သည် ခွင့်ပြုထားသော အများဆုံး အရွယ်အစား 10 MB ထက် ကျော်လွန်နေသည်။ ကျေးဇူးပြု၍ ပိုမိုသေးငယ်သော ဖိုင်ကို ရွေးချယ်ပါ။) ဟု ပြသပါ။ |
| `415` | `UNSUPPORTED_MEDIA_TYPE` | Upload လုပ်ထားသော ဖိုင်၏ MIME အမျိုးအစားသည် ခွင့်ပြုထားသော အစုအဝေးတွင် မပါဝင်ပါ။ | "The selected file type is not supported. Permitted types: PDF, PNG, JPG, JPEG." (ရွေးချယ်ထားသော ဖိုင်အမျိုးအစားကို ပံ့ပိုးမပေးပါ။ ခွင့်ပြုထားသော အမျိုးအစားများ: PDF, PNG, JPG, JPEG။) ဟု ပြသပါ။ |
| `422` | `UNPROCESSABLE_ENTITY` | စီးပွားရေးဆိုင်ရာ စည်းမျဉ်း ချိုးဖောက်မှု (ဥပမာ၊ ပြေစာ လိုအပ်သော်လည်း ပူးတွဲမထားခြင်း၊ မမှန်ကန်သော အခြေအနေ ပြောင်းလဲရန် ကြိုးပမ်းခြင်း)။ | တိကျသော စီးပွားရေးဆိုင်ရာ စည်းမျဉ်း ချိုးဖောက်မှု မက်ဆေ့ဂျ်ကို ပြသပါ။ |
| `500` | `INTERNAL_SERVER_ERROR` | မျှော်လင့်မထားသော ဆာဗာ အမှား။ | ယေဘုယျ အမှား မက်ဆေ့ဂျ်ကို ပြသပါ: "An unexpected error occurred. Please try again or contact your system administrator." (မျှော်လင့်မထားသော အမှားတစ်ခု ဖြစ်ပေါ်ခဲ့သည်။ ကျေးဇူးပြု၍ ပြန်လည်ကြိုးစားပါ သို့မဟုတ် သင့်စနစ် စီမံခန့်ခွဲသူကို ဆက်သွယ်ပါ။) ဆာဗာဘက်တွင် အပြည့်အစုံ stack trace ကို မှတ်တမ်းတင်ပါ။ |

### 7.3 အရှေ့ပိုင်း (Frontend) အမှား ပြသမှု အပြုအမူ

| အမှား အမျိုးအစား | ပြသသည့် နည်းလမ်း |
| :--- | :--- |
| **အကွက်-အဆင့် အတည်ပြုစစ်ဆေးမှု အမှား** | မမှန်ကန်သော ထည့်သွင်းမှု အကွက်တွင် အနီရောင် ဘောင်။ အမှား မက်ဆေ့ဂျ် စာသားကို အကွက်၏ အောက်တွင် အနီရောင်ဖြင့် တိုက်ရိုက်ပြသသည်။ အကွက်ကို မပြင်ဆင်မချင်း အာရုံစိုက်မှု (focus) ဆက်လက်ရှိနေမည်။ |
| **ပုံစံ-အဆင့် အတည်ပြုစစ်ဆေးမှု အကျဉ်းချုပ်** | ပုံစံ၏ အပေါ်ဆုံးတွင် ပြသထားသော ပိတ်၍ရသည့် အမှား ဘန်နာ (banner) တစ်ခု၊ ၎င်းတွင် သက်ဆိုင်ရာ အကွက်များဆီသို့ လင့်ခ်များ (anchor links) ပါဝင်ပြီး အတည်ပြုစစ်ဆေးမှု အမှားများအားလုံးကို စာရင်းပြုစုထားသည်။ |
| **API အမှား (အတည်ပြုစစ်ဆေးမှု-မဟုတ်သော)** | Viewport ၏ အပေါ်-ညာဘက် ထောင့်တွင် ပြသထားသော Toast အသိပေးချက်။ ၈ စက္ကန့်အကြာတွင် အလိုအလျောက် ပိတ်သွားမည်။ ကိုယ်တိုင် ပိတ်ရန် ရွေးချယ်မှု ရရှိနိုင်သည်။ အရောင်ဖြင့် ခွဲခြားထားသည်: အမှားများအတွက် အနီရောင်၊ သတိပေးချက်များအတွက် အဝါရောင် (amber)။ |
| **ကွန်ရက် အမှား / အချိန်လွန်သွားခြင်း (Timeout)** | အကျယ်အပြည့် ဘန်နာ: "Unable to connect to the server. Please check your network connection and try again." (ဆာဗာသို့ ချိတ်ဆက်၍ မရပါ။ ကျေးဇူးပြု၍ သင့်ကွန်ရက် ချိတ်ဆက်မှုကို စစ်ဆေးပြီး ပြန်လည်ကြိုးစားပါ။) ပြန်လည်ကြိုးစားရန် (Retry) ခလုတ် ထောက်ပံ့ပေးထားသည်။ |

## 8. ခွင့်ပြုချက်နှင့် ဝင်ရောက်ခွင့် ထိန်းချုပ်မှု

### 8.1 အထောက်အထား စစ်ဆေးခြင်း လိုအပ်ချက်များ

| ဂုဏ်သတ္တိ | သတ်မှတ်ချက် |
| :--- | :--- |
| **အထောက်အထား စစ်ဆေးခြင်း ယန္တရား** | JSON Web Token (JWT) Bearer Token |
| **တိုကင် သယ်ယူပို့ဆောင်ရေး** | API တောင်းဆိုမှုတိုင်းတွင် `Authorization: Bearer <token>` HTTP ခေါင်းစဉ် |
| **တိုကင် အတည်ပြုစစ်ဆေးခြင်း** | ဝင်လာသော တောင်းဆိုမှုတိုင်းတွင် တိုကင်၏ လက်မှတ် (signature)၊ သက်တမ်းကုန်ဆုံးချိန် (`exp` claim)၊ နှင့် ထုတ်ပေးသူ (`iss` claim) တို့ကို ဆာဗာဘက်မှ စစ်ဆေးအတည်ပြုခြင်း။ |
| **ဝင်ရောက်မှု အကြိမ် (Session) စီမံခန့်ခွဲမှု** | ပိုမိုကောင်းမွန်သော လုံခြုံရေးအတွက် Redis ကို အသုံးပြု၍ ဝင်ရောက်မှု အကြိမ် မှတ်တမ်း (session caching) ပြုလုပ်ထားသော Stateless JWT။ ဝင်ရောက်မှု အကြိမ် TTL: ၃,၆၀၀ စက္ကန့် (၁ နာရီ၊ sliding window)။ |
| **တိုကင် အသစ်ပြန်လည်ရယူခြင်း (Refresh)** | သက်တမ်းမကုန်ဆုံးမီ ကလိုင်းယင့်ဘက်မှ တိုကင် အသစ်ပြန်လည်ရယူခြင်း။ ဝင်ရောက်မှု အကြိမ် တစ်ဝက်တစ်ပျက်တွင် တိုကင် သက်တမ်းကုန်သွားပါက၊ အသုံးပြုသူအား ဝင်ရောက်မှု အကြိမ် သက်တမ်းကုန်ဆုံးကြောင်း အသိပေးချက်နှင့်အတူ ဝင်ရောက်ရန် မျက်နှာပြင်သို့ ပြန်လည်လမ်းညွှန်မည်။ |

### 8.2 အခွင့်အာဏာဆိုင်ရာ အစောင့် (Authorization Guard) တည်ဆောက်ပုံ

လျှောက်ထားသူ၏ ဒက်ရှ်ဘုတ် API endpoint တိုင်းတွင် အောက်ပါ NestJS အစောင့်များကို အစဉ်လိုက် အသုံးပြုထားသည်-

| အစောင့် အစီအစဉ် | အစောင့် အမည် | ရည်ရွယ်ချက် | ကျရှုံးပါက တုံ့ပြန်မှု |
| :---: | :--- | :--- | :--- |
| 1 | `JwtAuthGuard` | JWT Bearer တိုကင်ကို အတည်ပြုစစ်ဆေးသည်။ တောင်းဆိုမှု object တွင် အထောက်အထား စစ်ဆေးပြီးသော အသုံးပြုသူ၏ အကြောင်းအရာကို ထုတ်ယူပြီး ပူးတွဲပေးသည်။ | HTTP `401 Unauthorized` |
| 2 | `RolesGuard` | အထောက်အထား စစ်ဆေးပြီးသော အသုံးပြုသူ၏ `role_id` သည် `APPLICANT` အခန်းကဏ္ဍ ကုဒ်နှင့် ကိုက်ညီကြောင်း အတည်ပြုစစ်ဆေးသည်။ လျှောက်ထားသူမဟုတ်သော အခန်းကဏ္ဍရှိ အသုံးပြုသူများထံမှ တောင်းဆိုမှုများကို ပယ်ချသည်။ | HTTP `403 Forbidden` |
| 3 | `OwnershipGuard` | အရင်းအမြစ်-တိကျသော endpoint များအတွက် (ဥပမာ၊ `GET /api/payment-requests/:id`)၊ `payment_requests.applicant_user_id` သည် အထောက်အထား စစ်ဆေးပြီးသော အသုံးပြုသူ၏ `user_id` နှင့် ကိုက်ညီကြောင်း အတည်ပြုစစ်ဆေးသည်။ | HTTP `403 Forbidden` |

### 8.3 API Endpoint ခွင့်ပြုချက် မက်ထရစ် (Matrix)

| Endpoint | Method | လိုအပ်သော အခန်းကဏ္ဍ | ပိုင်ဆိုင်မှု စစ်ဆေးခြင်း | ဖော်ပြချက် |
| :--- | :---: | :--- | :---: | :--- |
| `/api/payment-requests/my-requests` | `GET` | `APPLICANT` | သွယ်ဝိုက်သော (မေးမြန်းမှု စစ်ထုတ်ခြင်း) | ပိုင်ဆိုင်သော ငွေပေးချေမှု တောင်းခံလွှာများအားလုံးကို စာရင်းပြုစုရန်။ |
| `/api/payment-requests` | `POST` | `APPLICANT` | N/A (အရင်းအမြစ်အသစ်) | ငွေပေးချေမှု တောင်းခံလွှာ မူကြမ်းအသစ် ဖန်တီးရန်။ |

### 8.4 လုံခြုံရေး စာရင်းစစ် မှတ်တမ်းတင်ခြင်း (Security Audit Logging)

လျှောက်ထားသူမှ ပြုလုပ်သော အခြေအနေ ပြောင်းလဲစေသည့် လုပ်ဆောင်ချက်တိုင်းကို `approval_logs` ဇယားတွင် အောက်ပါ မက်တာဒေတာ (metadata) များဖြင့် မှတ်တမ်းတင်ထားသည်-

| အကွက် | အရင်းအမြစ် | ရည်ရွယ်ချက် |
| :--- | :--- | :--- |
| `action_taken_by_user_id` | JWT `user_id` claim | လုပ်ဆောင်ချက်ကို ပြုလုပ်ခဲ့သော သရုပ်ဆောင်ကို ခွဲခြားသတ်မှတ်ရန်။ |
| `action_type_id` | စီးပွားရေး ယုတ္တိဗေဒ မြေပုံဆွဲခြင်း (Business logic mapping) | လုပ်ဆောင်ချက် အမျိုးအစားကို ခွဲခြားသတ်မှတ်ရန် (CREATED, EDITED, SUBMITTED စသဖြင့်)။ |
| `previous_status_id` | ပြောင်းလဲမှုမတိုင်မီ လက်ရှိ `status_id` | စာရင်းစစ် မှတ်တမ်း ပြန်လည်တည်ဆောက်ရန်အတွက် မူလ အခြေအနေကို မှတ်တမ်းတင်ရန်။ |
| `new_status_id` | ပြောင်းလဲမှုအပြီး ပစ်မှတ် `status_id` | သွားရောက်မည့် အခြေအနေကို မှတ်တမ်းတင်ရန်။ |
| `comment` | အသုံးပြုသူ ထည့်သွင်းချက် (သက်ဆိုင်ပါက) | လျှောက်ထားသူ၏ မှတ်ချက်များ သို့မဟုတ် စနစ်မှ-ဖန်တီးထားသော မှတ်စုများကို မှတ်တမ်းတင်ရန်။ |
| `ip_address` | တောင်းဆိုမှု မက်တာဒေတာ (`X-Forwarded-For` သို့မဟုတ် socket address) | လုံခြုံရေး စာရင်းစစ် — ကလိုင်းယင့် ကွန်ရက် ဇာစ်မြစ်ကို ခွဲခြားသတ်မှတ်ရန်။ |
| `user_agent` | တောင်းဆိုမှု ခေါင်းစဉ် `User-Agent` | လုံခြုံရေး စာရင်းစစ် — ကလိုင်းယင့် ဘရောက်ဇာ/စက်ပစ္စည်းကို ခွဲခြားသတ်မှတ်ရန်။ |
| `timestamp` | ဆာဗာမှ-ဖန်တီးသော `CURRENT_TIMESTAMP` (UTC) | လုပ်ဆောင်ချက်၏ တိကျသော အချိန်မှတ်တမ်း။ |

`approval_logs` ဇယားကို `trg_approval_logs_immutable` PostgreSQL trigger ဖြင့် ကာကွယ်ထားပြီး၊ ယင်းသည် `UPDATE` နှင့် `DELETE` လုပ်ဆောင်မှုများအားလုံးကို တားဆီးသည်။ ဤဇယားရှိ မှတ်တမ်းများသည် ပေါင်းထည့်ရုံသာ (append-only) ဖြစ်ပြီး၊ စည်းမျဉ်းစည်းကမ်း လိုက်နာမှု လိုအပ်ချက်များအရ အနည်းဆုံး ၅ နှစ် (၅) နှစ် ထိန်းသိမ်းထားရမည်။

## 9. အချိန်နှင့်တစ်ပြေးညီ အသိပေးချက် အပြုအမူ

### 9.1 WebSocket ဖွဲ့စည်းမှု

| ဂုဏ်သတ္တိ | သတ်မှတ်ချက် |
| :--- | :--- |
| **နည်းပညာ** | Socket.IO (NestJS `@WebSocketGateway` မှတစ်ဆင့်) |
| **Gateway Port** | 3001 |
| **CORS မူဝါဒ** | ပတ်ဝန်းကျင်အလိုက် ဖွဲ့စည်းသတ်မှတ်နိုင်သည်။ Development: `origin: '*'`. Production: အက်ပလီကေးရှင်း ဒိုမိန်းသို့ ကန့်သတ်ထားသည်။ |
| **ချိတ်ဆက်မှု ပရိုတိုကော** | ဝင်ရောက်သည့်အခါ၊ အရှေ့ပိုင်း ကလိုင်းယင့်သည် WebSocket ချိတ်ဆက်မှုကို တည်ဆောက်ပြီး `{ role: 'APPLICANT', userId: <user_id> }` ဖြင့် `joinRoom` ဖြစ်ရပ်ကို ထုတ်လွှင့်သည်။ |
| **အခန်း အဖွဲ့ဝင်ဖြစ်မှု** | လျှောက်ထားသူတစ်ဦးစီသည် အခန်းနှစ်ခုသို့ ဝင်ရောက်သည်: (၁) ၎င်းတို့၏ အခန်းကဏ္ဍ အခန်း (`'APPLICANT'`)၊ နှင့် (၂) ၎င်းတို့၏ ကိုယ်ပိုင် အခန်း (`'user:<user_id>'`)။ |

### 9.2 လျှောက်ထားသူမှ လက်ခံရရှိသော အသိပေးချက် ဖြစ်ရပ်များ

| ဖြစ်ရပ် အမည် | အစပျိုးသည့် အခြေအနေ | Payload | UI အပြုအမူ |
| :--- | :--- | :--- | :--- |
| `statusUpdate` | ပိုင်ဆိုင်သော တောင်းခံလွှာတစ်ခု၏ အခြေအနေ ပြောင်းလဲသွားသည် (ဥပမာ၊ မန်နေဂျာ အတည်ပြုသည်၊ မန်နေဂျာ ပယ်ချသည်၊ အတည်ပြုသူ အတည်ပြုသည်၊ စာရင်းကိုင်မှ ငွေပေးချေပြီးကြောင်း မှတ်သားသည်)။ | `{ payment_request_id, request_number, previous_status, new_status, action_by, timestamp }` | 1. အခြေအနေ ပြောင်းလဲမှု အကျဉ်းချုပ်နှင့်အတူ toast အသိပေးချက်ကို ပြသပါ။ 2. အခြေအနေ တံဆိပ် (badge) အသစ်ဖြင့် ငွေပေးချေမှု တောင်းခံလွှာ စာရင်းရှိ သက်ရောက်မှုရှိသော အတန်းကို အသစ်ပြင်ဆင်ပါ။ 3. သက်ရောက်မှုရှိသော တောင်းခံလွှာ၏ အသေးစိတ် မြင်ကွင်းကို လက်ရှိ ဖွင့်ထားပါက၊ အခြေအနေ ပြသမှုနှင့် အတည်ပြုချက် အချိန်ဇယားကို refresh လုပ်ပါ။ |
| `notification` | တိကျသော အသုံးပြုသူကို ပစ်မှတ်ထားသည့် ကိုယ်ပိုင် အသိပေးချက် (ဥပမာ၊ ပယ်ချသည့် မှတ်ချက် အသိပေးချက်)။ | `{ type, title, message, payment_request_id, timestamp }` | 1. ထင်ရှားသော toast အသိပေးချက်ကို ပြသပါ။ 2. မဖတ်ရသေးသော အသိပေးချက် တံဆိပ် (badge) ကောင်တာကို တိုးပါ။ |

### 9.3 လျှောက်ထားသူ၏ လုပ်ဆောင်ချက်များမှ ပေးပို့သော အသိပေးချက် ဖြစ်ရပ်များ

| လျှောက်ထားသူ လုပ်ဆောင်ချက် | ပစ်မှတ် အခန်း | ဖြစ်ရပ် အမည် | Payload |
| :--- | :--- | :--- | :--- |
| မန်နေဂျာထံ တင်ပြခြင်း | `user:<manager_user_id>` + `MANAGER` အခန်းကဏ္ဍ အခန်း | `statusUpdate` | `{ payment_request_id, request_number, new_status: 'SUBMITTED_MANAGER', submitted_by: <applicant_name>, timestamp }` |

### 9.4 ချိတ်ဆက်မှု ခံနိုင်ရည်ရှိမှု (Connection Resilience)

| အခြေအနေ | အပြုအမူ |
| :--- | :--- |
| **ယာယီ ချိတ်ဆက်မှု ပြတ်တောက်ခြင်း** | Socket.IO သည် exponential backoff ဖြင့် အလိုအလျောက် ပြန်လည်ချိတ်ဆက်သည်။ ဆိုင်းငံ့ထားသော ဖြစ်ရပ်များကို တန်းစီထားပြီး ပြန်လည်ချိတ်ဆက်သည့်အခါ ပေးပို့သည်။ |
| **ကြာရှည်စွာ ချိတ်ဆက်မှု ပြတ်တောက်ခြင်း** | ပြန်လည်ချိတ်ဆက်သည့်အခါ၊ အရှေ့ပိုင်းသည် နောက်ဆုံး ဆာဗာ အခြေအနေနှင့် ငွေပေးချေမှု တောင်းခံလွှာ စာရင်းကို ထပ်တူကျစေရန် (synchronize) ဒေတာ အပြည့်အစုံ refresh တောင်းဆိုမှုကို ပြုလုပ်သည်။ |
| **ကြာမြင့်ချိန် လိုအပ်ချက် (Latency Requirement)** | ဆာဗာဘက်မှ လွှဲပြောင်းမှု အတည်ပြုပြီးနောက် (NFR-003 အရ) ၅၀၀ မီလီစက္ကန့်အတွင်း အခြေအနေ ပြောင်းလဲမှု အသိပေးချက်များကို ချိတ်ဆက်ထားသော ကလိုင်းယင့်များထံ ပေးပို့ရမည်။ |

### 10.1 အဝင် လမ်းကြောင်းရှာဖွေခြင်း (Inbound Navigation)

| မူလ မျက်နှာပြင် | လမ်းကြောင်းရှာဖွေခြင်း လမ်းကြောင်း | အခြေအနေ |
| :--- | :--- | :--- |

| အထွေထွေ လမ်းကြောင်းပြ ခေါင်းစဉ် | အပေါ်ဆုံး လမ်းကြောင်းပြ ဘားရှိ "申請者" (Applicant) လင့်ခ်ကို နှိပ်ခြင်း။ | အသုံးပြုသူကို အထောက်အထား စစ်ဆေးပြီးဖြစ်သည်။ |

## 11. လုပ်ဆောင်ချက်မဟုတ်သော ထည့်သွင်းစဉ်းစားမှုများ

### 11.1 စွမ်းဆောင်ရည် လိုအပ်ချက်များ

| မက်ထရစ် (Metric) | ပစ်မှတ် | တိုင်းတာမှု နည်းလမ်း |
| :--- | :--- | :--- |
| Dashboard စာမျက်နှာ ဖွင့်ချိန် (ကနဦး render) | ≤ ၂ စက္ကန့် | လမ်းကြောင်းရှာဖွေခြင်း စတင်ချိန်မှစ၍ ဒေတာများ ဖြည့်သွင်းပြီး DOM အပြည့်အစုံ ရေးဆွဲချိန် (paint) အထိ အချိန်။ |
| ငွေပေးချေမှု တောင်းခံလွှာ စာရင်း မေးမြန်းမှု တုံ့ပြန်မှု | ≤ ၁ စက္ကန့် | ပုံသေ ဘောင်များပါရှိသော စာမျက်နှာခွဲခြားထားသည့် စာရင်းအတွက် ဆာဗာဘက်မှ API တုံ့ပြန်ချိန်။ `idx_payment_requests_applicant_id` နှင့် `idx_payment_requests_status_created` အညွှန်းများ (indexes) ဖြင့် အကောင်းဆုံးဖြစ်အောင် ပြုလုပ်ထားသည်။ |
| မူကြမ်း သိမ်းဆည်းမှု လုပ်ဆောင်ချက် ကြာချိန် (round-trip) | ≤ ၂ စက္ကန့် | ပုံစံ တင်ပြချိန်မှစ၍ ဆာဗာ အတည်ပြုချက်နှင့် UI အသစ်ပြင်ဆင်ချိန် အထိ အချိန်။ |
| ပြေစာ ဖိုင် upload | 10 MB ဖိုင်အတွက် ≤ ၅ စက္ကန့် | Upload စတင်ချိန်မှစ၍ ဆာဗာ အတည်ပြုချက် အထိ အချိန်၊ ကွန်ရက် bandwidth အပေါ် မူတည်သည်။ |
| WebSocket အသိပေးချက် ပေးပို့မှု | ≤ ၅၀၀ မီလီစက္ကန့် | ဆာဗာဘက်မှ လွှဲပြောင်းမှု အတည်ပြုချိန်မှစ၍ ကလိုင်းယင့်ဘက်မှ toast ပြသချိန် အထိ အချိန်။ |

### 11.2 ကက်ရှ် (Caching) မဟာဗျူဟာ

| Cache နယ်ပယ် | သော့ချက် ပုံစံ (Key Pattern) | TTL | ပယ်ဖျက်ရန် အစပျိုးခြင်း |
| :--- | :--- | :--- | :--- |
| မာစတာ အချက်အလက် ဇယားများ (ငွေကြေးများ၊ ငွေပေးချေမှု အမျိုးအစားများ၊ ငွေပေးချေမှု နည်းလမ်းများ၊ အခြေအနေများ) | `lookup:<table_name>` | ၈၆,၄၀၀ စက္ကန့် (၂၄ နာရီ) | မာစတာ ဒေတာကို စီမံခန့်ခွဲသူမှ ပြင်ဆင်ခြင်း။ |
| အသက်ဝင်နေသော ငွေပေးချေမှု တောင်းခံလွှာ payload | `payment_request:payload:<id>` | ၆၀၀ စက္ကန့် (၁၀ မိနစ်) | မည်သည့် အခြေအနေ ပြောင်းလဲမှု၊ အကွက် အသစ်ပြင်ဆင်မှု၊ သို့မဟုတ် တောင်းခံလွှာပေါ်ရှိ လုပ်ဆောင်ချက်မဆို။ လွှဲပြောင်းမှု အတည်ပြုရာတွင် `DEL` မှတစ်ဆင့် ဖယ်ရှားသည်။ |
| မန်နေဂျာ အသုံးပြုသူ စာရင်း (dropdown အတွက်) | `lookup:active_managers` | ၃,၆၀၀ စက္ကန့် (၁ နာရီ) | အသုံးပြုသူ အခန်းကဏ္ဍများ သို့မဟုတ် အသက်ဝင်သော အခြေအနေများကို စီမံခန့်ခွဲသူမှ ပြောင်းလဲခြင်း။ |

### 11.3 တုံ့ပြန်မှုရှိသော ဒီဇိုင်း (Responsive Design) လိုအပ်ချက်များ

လျှောက်ထားသူ၏ ဒက်ရှ်ဘုတ် ဖွဲ့စည်းပုံသည် အောက်ပါ စက်ပစ္စည်း အမျိုးအစားများတစ်လျှောက် တုံ့ပြန်မှုရှိပြီး အလုပ်လုပ်နိုင်ရမည်-

| စက်ပစ္စည်း အမျိုးအစား | Viewport အတိုင်းအတာ | ဖွဲ့စည်းပုံ လိုက်လျောညီထွေဖြစ်မှု |
| :--- | :--- | :--- |
| Desktop | ≥ 1024px | ကော်လံနှစ်ခု ဖွဲ့စည်းပုံ: ဘယ်ဘက်တွင် တောင်းခံလွှာ စာရင်း/အချိန်ဇယား၊ ညာဘက်တွင် ပုံစံ/အသေးစိတ် မြင်ကွင်း။ |
| Tablet | 768px – 1023px | ခေါက်သိမ်းနိုင်သော အပိုင်းများပါဝင်သည့် ကော်လံတစ်ခုတည်း စီထားသော ဖွဲ့စည်းပုံ။ |
| Mobile | < 768px | အကော်ဒီယံပုံစံ (accordion-style) ပုံစံ အပိုင်းများနှင့် အောက်ခြေ-စာရွက် (bottom-sheet) လုပ်ဆောင်ချက် ခလုတ်များ ပါဝင်သည့် အကျယ်အပြည့် ကော်လံတစ်ခုတည်း ဖွဲ့စည်းပုံ။ |
