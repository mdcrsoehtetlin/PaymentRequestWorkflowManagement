## 1. ပရောဂျက် ခြုံငုံသုံးသပ်ချက်နှင့် နောက်ခံသမိုင်း (Project Overview & Background)

### 1.1 ပရောဂျက်အမည် (Project Name)
**ငွေပေးချေမှု တောင်းဆိုခြင်း လုပ်ငန်းအဆင့်ဆင့် စီမံခန့်ခွဲမှုစနစ် (Payment Request Workflow Management System)** (支払申請ワークフロー管理システム)

### 1.2 ရည်ရွယ်ချက်နှင့် ပန်းတိုင်များ (Purpose & Objectives)
ဤစနစ်သည် အဖွဲ့အစည်းအတွင်းရှိ ဝန်ထမ်းများ၏ ငွေပေးချေမှု တောင်းဆိုခြင်းများအား စတင်လျှောက်ထားချိန်မှစ၍ မန်နေဂျာမှ စိစစ်ခြင်း၊ နောက်ဆုံးအတည်ပြုသူမှ ခွင့်ပြုခြင်းနှင့် စာရင်းကိုင်မှ ငွေပေးချေမှု ပြုလုပ်ခြင်းအထိ လုပ်ငန်းစဉ်တစ်ခုလုံးကို စီမံခန့်ခွဲပေးမည်ဖြစ်သည်။ စနစ်သည် ပွင့်လင်းမြင်သာမှုရှိသော ခြေရာခံခြင်း၊ သင့်လျော်သော အခွင့်အာဏာထိန်းချုပ်မှုနှင့် ငွေပေးချေမှု တောင်းဆိုလွှာအခြေအနေကို အချိန်နှင့်တပြေးညီ သိရှိနိုင်မှုကို သေချာစေသည်။

### 1.3 စီးပွားရေးဆိုင်ရာ အကြောင်းအရာ (Business Context)
- **ပြဿနာဖော်ပြချက် (Problem Statement):** လက်ဖြင့် (Manual) ငွေပေးချေမှု တောင်းဆိုခြင်းကို ခြေရာခံခြင်းသည် မှားယွင်းမှုများရှိနိုင်ခြင်း၊ ပွင့်လင်းမြင်သာမှု မရှိခြင်းနှင့် ငွေပေးချေမှု လုပ်ငန်းစဉ်များကို နှောင့်နှေးစေသည်။ အတည်ပြုသူများနှင့် လျှောက်ထားသူများသည် တောင်းဆိုလွှာ၏ အခြေအနေကို အချိန်နှင့်တပြေးညီ ခြေရာမခံနိုင်ပါ။
- **ဖြေရှင်းမည့် နည်းလမ်း (Solution Approach):** အလိုအလျောက် အခြေအနေပြောင်းလဲမှုများ၊ WebSocket မှတစ်ဆင့် အချိန်နှင့်တပြေးညီ အသိပေးချက်များနှင့် ရှင်းလင်းသော စာရင်းစစ်မှတ်တမ်း (audit trail) တို့ပါဝင်သည့် တစ်နေရာတည်းမှ စီမံခန့်ခွဲနိုင်သော၊ ကဏ္ဍအလိုက် (role-based) လုပ်ငန်းအဆင့်ဆင့်စနစ်ကို အကောင်အထည်ဖော်ရန်ဖြစ်သည်။
- **မျှော်မှန်းထားသော ရလဒ်များ (Expected Outcomes):**
  - ငွေပေးချေမှု လုပ်ငန်းစဉ်ကြာချိန်ကို လျှော့ချနိုင်ခြင်း
  - သက်ဆိုင်သူအားလုံးအတွက် ပွင့်လင်းမြင်သာမှု တိုးတက်လာခြင်း
  - အချိန်နှင့်တပြေးညီ အပ်ဒိတ်များဖြင့် အလိုအလျောက် အခြေအနေ ခြေရာခံနိုင်ခြင်း
  - သင့်လျော်သော မှတ်တမ်းတင်ခြင်းနှင့် ငွေပေးချေမှု ခွင့်ပြုချက် စည်းမျဉ်းများကို လိုက်နာမှုရှိခြင်း

### 1.4 ပရောဂျက် ဘောင်သတ်မှတ်ချက် (Project Scope)
- **ပါဝင်သည့်အရာများ (Included):** အဓိက လုပ်ငန်းအဆင့်ဆင့် စီမံခန့်ခွဲမှု၊ ကဏ္ဍအလိုက် အသုံးပြုသူ အကောင့်ဝင်ရောက်မှု/ခွင့်ပြုချက်များ၊ ငွေပေးချေမှု တောင်းဆိုခြင်း သက်တမ်းတစ်ခုလုံး စီမံခန့်ခွဲမှု၊ အချိန်နှင့်တပြေးညီ အခြေအနေ ခြေရာခံမှု၊ ပြေစာဖိုင် စီမံခန့်ခွဲမှု၊ စာရင်းကိုင် dashboard။
- **မပါဝင်သည့်အရာများ (Excluded):** ပြင်ပဘဏ် API များနှင့် ချိတ်ဆက်ခြင်း၊ အလိုအလျောက် ငွေပေးချေမှု ပြုလုပ်ခြင်း၊ အဆင့်မြင့်အစီရင်ခံစာများ/ခွဲခြမ်းစိတ်ဖြာမှုများ၊ ဘာသာစကားအမျိုးမျိုး အသုံးပြုနိုင်မှု။

### 1.5 နည်းပညာများ (Technology Stack)
- **Backend:** TypeScript + NestJS (REST APIs)
  - Framework: NestJS နှင့် Express.js
  - Language: TypeScript
  - API Pattern: RESTful JSON APIs
- **Frontend:** React (PC/Smartphone တွဲဖက်သုံးနိုင်သော UI)
  - UI Framework: React (Responsive design ဖြင့်)
  - Compatibility: Desktop နှင့် mobile browser များ
- **Database:** PostgreSQL (Primary) + Redis (Caching/Session Management)
  - Primary Database: PostgreSQL (အမြဲတမ်း သိမ်းဆည်းရန်)
  - Cache/Session: Redis (လျင်မြန်သော session စီမံခန့်ခွဲမှုနှင့် caching အတွက်)
  - ORM: TypeORM သို့မဟုတ် Prisma
- **Real-Time Communication:** WebSocket (Socket.io) အချိန်နှင့်တပြေးညီ အသိပေးချက်များအတွက်
- **File Storage:** Local file system abstraction layer (အနာဂတ်တွင် cloud သို့ပြောင်းရွှေ့ရန် ရည်ရွယ်ဒီဇိုင်းဆွဲထားသည်)

## 2. အသုံးပြုသူ ကဏ္ဍများနှင့် ခွင့်ပြုချက်များ (User Roles & Permissions)

### 2.1 အသုံးပြုသူ ကဏ္ဍများ ခြုံငုံသုံးသပ်ချက် (User Roles Overview)

| ကဏ္ဍ (Role) | ဂျပန်အမည် (Japanese Name) | အဓိက တာဝန်ယူမှု (Primary Responsibility) | အဓိက လုပ်ပိုင်ခွင့်များ (Key Permissions) |
|------|---------------|----------------------|-----------------|
| **Applicant** (လျှောက်ထားသူ) | 申請者 | ငွေပေးချေမှု တောင်းဆိုလွှာများ တင်သွင်းရန်နှင့် ၎င်းတို့၏ တောင်းဆိုလွှာများကို စီမံရန် | • ငွေပေးချေမှု တောင်းဆိုလွှာ ဖန်တီးရန်<br>• မိမိ၏ Draft တောင်းဆိုလွှာများကို ပြင်ဆင်ရန်<br>• ပြေစာဖိုင်များ တင်ရန်<br>• မိမိ၏ တောင်းဆိုလွှာ မှတ်တမ်းကို ကြည့်ရှုရန်<br>• ငြင်းပယ်ခံရပါက ပြန်လည်တင်ပြရန်<br>• (Manager မှ အတည်ပြုပြီးပါက) နောက်ဆုံးအတည်ပြုသူထံ တင်ပြရန်<br>• တုံ့ပြန်ချက်များကို အကြောင်းပြန်ရန် |
| **Manager** (မန်နေဂျာ) | 担当マネージャー | ငွေပေးချေမှု တောင်းဆိုလွှာများကို ပထမအဆင့် စိစစ်အတည်ပြုရန် | • သတ်မှတ်ပေးထားသော တောင်းဆိုလွှာများကို ကြည့်ရှုရန်<br>• တောင်းဆိုလွှာများကို စိစစ်အတည်ပြု (approve) ရန်<br>• မှတ်ချက်များဖြင့် တောင်းဆိုလွှာများကို ငြင်းပယ် (reject) ရန်<br>• လျှောက်ထားသူထံ တောင်းဆိုလွှာများကို ပြန်လည်ပေးပို့ရန်<br>• တောင်းဆိုလွှာ အသေးစိတ်နှင့် ပူးတွဲဖိုင်များကို ကြည့်ရှုရန် |
| **Final Approver** (နောက်ဆုံးအတည်ပြုသူ) | 最終承認者 | နောက်ဆုံး ခွင့်ပြုချက် ဆုံးဖြတ်ချက်ပေးရန် | • အတည်ပြုရန်အဆင့်ရှိ တောင်းဆိုလွှာများကို ကြည့်ရှုရန်<br>• တောင်းဆိုလွှာများကို အတည်ပြုရန်<br>• မှတ်ချက်များဖြင့် တောင်းဆိုလွှာများကို ငြင်းပယ်ရန်<br>• လျှောက်ထားသူထံ တောင်းဆိုလွှာများကို ပြန်လည်ပေးပို့ရန်<br>• စာရင်းစစ်မှတ်တမ်း (audit trail) ကို ကြည့်ရှုရန် |
| **Accounting** (စာရင်းကိုင်) | 経理 | အတည်ပြုပြီးသော ငွေပေးချေမှုများကို ပြုလုပ်ရန် | • အတည်ပြုပြီးသော ငွေပေးချေမှု စောင့်ဆိုင်းစာရင်းကို ကြည့်ရှုရန်<br>• ငွေပေးချေမှု ပြီးစီးကြောင်း မှတ်သားရန်<br>• ဌာနခွဲအလိုက် အထူးဆောင်ရွက်ရမည့် မှတ်စုများကို ကြည့်ရှုရန်<br>• ငွေပေးချေမှု အစီရင်ခံစာများ ထုတ်ရန်<br>• အရောင်းအဝယ်မှတ်တမ်းကို ကြည့်ရှုရန် |
| **System Administrator** (စနစ် စီမံခန့်ခွဲသူ) | システム管理者 | စနစ်ပြင်ဆင်သတ်မှတ်မှုများနှင့် အသုံးပြုသူများကို စီမံခန့်ခွဲရန် | • User အကောင့်များနှင့် ကဏ္ဍများကို စီမံရန်<br>• စနစ်ပြင်ဆင်မှုများကို သတ်မှတ်ရန်<br>• Audit logs များကို ကြည့်ရှုရန် |

### 2.2 ကဏ္ဍအလိုက် လုပ်ပိုင်ခွင့် ထိန်းချုပ်မှု (Role-Based Access Control - RBAC)

ဝင်ရောက်ခွင့် ထိန်းချုပ်မှုကို controller နှင့် page အဆင့် နှစ်ခုလုံးတွင် ကျင့်သုံးထားသည်-

- **Applicant Dashboard:** မိမိ၏ တောင်းဆိုလွှာများကို ကြည့်ရှုရန်၊ အသစ်ဖန်တီးရန်၊ draft များကို ပြင်ဆင်ရန်၊ ပြေစာများ တင်ရန်၊ အတည်ပြုသူထံ တင်ပြရန်။
- **Manager Dashboard:** စိစစ်ရန် စောင့်ဆိုင်းနေသော တောင်းဆိုလွှာများကို ကြည့်ရှုရန်၊ စိစစ်အတည်ပြုခြင်းများ လုပ်ဆောင်ရန်၊ မှတ်ချက်များ ထည့်ရန်။
- **Final Approver Dashboard:** နောက်ဆုံးအတည်ပြုရန် စောင့်ဆိုင်းနေသော တောင်းဆိုလွှာများကို ကြည့်ရှုရန်၊ အတည်ပြု ဆုံးဖြတ်ချက်များ ပြုလုပ်ရန်၊ မှတ်ချက်များ ထည့်ရန်။
- **Accounting Dashboard:** အတည်ပြုပြီးသော တောင်းဆိုလွှာများကို ကြည့်ရှုရန်၊ ငွေပေးချေမှု ပြီးစီးကြောင်း လုပ်ဆောင်ရန်၊ အထူး ဌာနခွဲ သတိပေးချက်များကို ကြည့်ရှုရန်။
- **Admin Panel:** အသုံးပြုသူ စီမံခန့်ခွဲမှု၊ စနစ် ပြင်ဆင်သတ်မှတ်မှု၊ audit log စစ်ဆေးမှု။

## 3. လုပ်ဆောင်ချက်ဆိုင်ရာ လိုအပ်ချက်များ (Functional Requirements)

### 3.1 အဓိက Entity များနှင့် ဒေတာ မော်ဒယ် (Core Entities & Data Model)

စနစ်သည် အောက်ပါ အဓိက entity များကို စီမံခန့်ခွဲသည်-

#### 3.1.1 User Entity
စနစ်အသုံးပြုသူများကို ကဏ္ဍ (role) သတ်မှတ်ချက်များဖြင့် ကိုယ်စားပြုသည်။

**အချက်အလက်များ (Attributes):**
- UserID (Primary Key)
- Email
- FullName
- Department
- Branch (ဥပမာ- "Yangon", "Mandalay", "Naypyidaw")
- Role (Applicant, Manager, Final Approver, Accounting, Admin)
- IsActive
- CreatedDate
- ModifiedDate

#### 3.1.2 PaymentRequest Entity
ငွေပေးချေမှု တောင်းဆိုလွှာတစ်ခု၏ သက်တမ်းတစ်ခုလုံးကို ကိုယ်စားပြုသည်။ စံသတ်မှတ်ထားသော ငွေပေးချေမှု တောင်းဆိုလွှာ ဖောင် (Payment Request Form - 支払申請書) ကို ပုံစံယူထားသည်။

**အချက်အလက်များ (Attributes):**
- PaymentRequestID (Primary Key)
- ApplicantUserID (Foreign Key to User)
- ManagerUserID (Foreign Key to User, nullable)
- FinalApproverUserID (Foreign Key to User, nullable)
- AccountingUserID (Foreign Key to User, nullable)
- RequestNumber (Unique identifier, အလိုအလျောက်ထုတ်ပေးသည်; ဥပမာ- "PRF-2026-001")

**ငွေပေးချေမှု တောင်းဆိုလွှာ အချက်အလက်များ (Payment Request Information):**
- ApplicationDate (Date, မဖြစ်မနေလိုအပ်သည်) - 申請日: တောင်းဆိုလွှာ တင်သွင်းသည့်ရက်စွဲ
- DesiredPaymentDate (Date, မဖြစ်မနေလိုအပ်သည်) - 支払希望日: လျှောက်ထားသူမှ ငွေပေးချေစေလိုသော ရက်စွဲ
- TotalAmount (Decimal, မဖြစ်မနေလိုအပ်သည်) - 支払金額: စုစုပေါင်း ငွေပေးချေမည့် ပမာဏ
- CurrencyType (String, မဖြစ်မနေလိုအပ်သည်) - 通貨選択: ငွေကြေးအမျိုးအစား (ဥပမာ- "MMK", "USD")
- PaymentType (String, မဖြစ်မနေလိုအပ်သည်) - 支払タイプ: ငွေပေးချေမှု အမျိုးအစား (ဥပမာ- "Expense Reimbursement", "Service Payment", "Other")
- PaymentMethod (String, မဖြစ်မနေလိုအပ်သည်) - 支払方法: ငွေပေးချေမည့် နည်းလမ်း (ဥပမာ- "Bank Transfer", "Cash", "Check")

**လျှောက်ထားသူ အသေးစိတ် (Applicant Details - User မှ အလိုအလျောက် ဖြည့်ပေးသည်):**
- ApplicantEmployeeNumber (String, မဖြစ်မနေလိုအပ်သည်) - 社員番号: User profile မှ ဝန်ထမ်းနံပါတ်
- ApplicantFullName (String, မဖြစ်မနေလိုအပ်သည်) - 氏名: User profile မှ ဝန်ထမ်းအမည်
- ApplicantBranch (String, မဖြစ်မနေလိုအပ်သည်) - User profile မှ ဌာနခွဲ

**အသေးစိတ် ရှင်းတမ်းများ (Breakdown Details):**
- Purpose (String, မဖြစ်မနေလိုအပ်သည်) - 用途: ငွေပေးချေရသည့် ရည်ရွယ်ချက်/အသုံးပြုမှု
- BankAccountInfo (String, ရွေးချယ်နိုင်သည်) - 銀行口座・口座名/電話番号: ဘဏ်အကောင့်၊ အကောင့်ပိုင်ရှင်အမည်၊ သို့မဟုတ် လက်ငင်းငွေချေမှုအတွက် ဖုန်းနံပါတ်
- RequestContent (String, မဖြစ်မနေလိုအပ်သည်) - 支払申請内容: ငွေပေးချေမှု တောင်းဆိုချက် အကြောင်းအရာ၏ ဖော်ပြချက်
- HasReceipt (Boolean, မဖြစ်မနေလိုအပ်သည်) - 領収書の有無: ပြေစာ ပူးတွဲထားခြင်း ရှိ/မရှိ (yes/no)
- ReceiptFilesPaths (String, nullable) - ပြေစာဖိုင် လမ်းကြောင်းများ (ကော်မာဖြင့် ခြားထားသည်; ဖိုင်များစွာကို ပံ့ပိုးပေးသည်)
- PaymentBreakdownItems (Collection of PaymentBreakdownItem, မဖြစ်မနေလိုအပ်သည်) - 支払内訳: အကြောင်းအရာအလိုက် အသေးစိတ်ဇယား (Line-by-line breakdown table)

**အခြေအနေနှင့် လုပ်ငန်းအဆင့်ဆင့် (Status & Workflow):**
- Status (String, မဖြစ်မနေလိုအပ်သည်) - တရားဝင် status များအတွက် အပိုင်း 3.2 တွင် ကြည့်ပါ
- CurrentAssignedTo (UserID, nullable) - မည်သူက အရေးယူဆောင်ရွက်ရမည်ကို ပြသသည်

**အချိန်မှတ်တမ်းများ (Timestamps):**
- CreatedDate (DateTime)
- SubmittedToManagerDate (DateTime, nullable)
- ManagerVerificationDate (DateTime, nullable)
- SubmittedToApproverDate (DateTime, nullable)
- ApprovalDate (DateTime, nullable)
- PaymentCompletedDate (DateTime, nullable)
- ModifiedDate (DateTime)
- IsDeleted (Boolean, soft delete flag)

#### 3.1.3 PaymentBreakdownItem Entity
ငွေပေးချေမှု တောင်းဆိုလွှာ၏ အသေးစိတ်ရှင်းတမ်းဇယား (支払内訳) မှ စာကြောင်းတစ်ကြောင်းကို ကိုယ်စားပြုသည်။ တောင်းဆိုလွှာတစ်ခုစီတွင် breakdown items များစွာ ပါဝင်နိုင်သည် (ဖောင်ပုံစံအရ အများဆုံး စာကြောင်း ၁၅ ကြောင်းအထိ)။

**အချက်အလက်များ (Attributes):**
- PaymentBreakdownItemID (Primary Key)
- PaymentRequestID (Foreign Key to PaymentRequest)
- LineNumber (Integer, 1-15) - No: Breakdown ဇယားရှိ စာကြောင်းနံပါတ်
- ItemDate (Date, မဖြစ်မနေလိုအပ်သည်) - 日付: အသုံးစရိတ် ရက်စွဲ
- Description (String, မဖြစ်မနေလိုအပ်သည်) - 内容: အကြောင်းအရာ/ဖော်ပြချက်
- Amount (Decimal, မဖြစ်မနေလိုအပ်သည်) - ဤစာကြောင်းအတွက် ပမာဏ
- Quantity (Decimal, ရွေးချယ်နိုင်သည်) - အရေအတွက် (လိုအပ်ပါက)
- UnitPrice (Decimal, ရွေးချယ်နိုင်သည်) - ခုလတ်တန်ဖိုး (လိုအပ်ပါက)
- CreatedDate (DateTime)
- ModifiedDate (DateTime)

**တွက်ချက်ထားသော အကွက် (Calculated Field):**
- Total (Decimal, calculated) - 合計: ငွေပေးချေမှု တောင်းဆိုလွှာအတွက် စာကြောင်းများရှိ ပမာဏအားလုံး၏ စုစုပေါင်း

#### 3.1.4 ApprovalLog Entity
စစ်ဆေးရေး (audit) ရည်ရွယ်ချက်များအတွက် ငွေပေးချေမှု တောင်းဆိုလွှာများအပေါ် လုပ်ဆောင်ခဲ့သော အခြေအနေ အပြောင်းအလဲများနှင့် လုပ်ဆောင်ချက်များ အားလုံးကို ခြေရာခံသည်။

**အချက်အလက်များ (Attributes):**
- ApprovalLogID (Primary Key)
- PaymentRequestID (Foreign Key to PaymentRequest)
- ActionTakenByUserID (Foreign Key to User)
- ActionType (ဥပမာ- "Created", "Submitted", "ManagerReview", "ManagerVerified", "ManagerRejected", "ApproverReview", "Approved", "ApproverRejected", "PaymentCompleted", "Edited")
- PreviousStatus
- NewStatus
- Comment
- Timestamp
- IPAddress (လုံခြုံရေး စစ်ဆေးမှုအတွက်)

### 3.2 သက်တမ်းတစ်လျှောက် အခြေအနေများ (Lifecycle Statuses)

စနစ်သည် အောက်ပါ status များကို အတိအကျ ကျင့်သုံးသည်။ အပြောင်းအလဲအားလုံးသည် ရှင်းလင်းပြတ်သားပြီး ခြေရာခံနိုင်ရမည်-

| Status Code | Status Name (အခြေအနေအမည်) | Triggered By (ဖြစ်ပေါ်စေသူ) | Assigned To (တာဝန်ပေးအပ်ခံရသူ) | Can Edit (ပြင်ဆင်နိုင်သလား) | Description (ဖော်ပြချက်) |
|-------------|-------------|--------------|-------------|----------|-------------|
| 1 | **Draft** | System (on create) | Applicant | Yes | ကနဦး အခြေအနေ; လျှောက်ထားသူသည် တောင်းဆိုလွှာကို ရေးဆွဲနေသည် |
| 2 | **Submitted to Manager** | Applicant action | Manager | No | လျှောက်ထားသူမှ တင်သွင်းလိုက်သည်; မန်နေဂျာ၏ လုပ်ဆောင်မှုကို စောင့်ဆိုင်းနေသည် |
| 3 | **Manager Reviewing** | Manager opens request | Manager | No | မန်နေဂျာသည် လက်ရှိ စစ်ဆေးနေသည် (မန်နေဂျာမှ ကြည့်ရှုသောအခါ အလိုအလျောက်ပြောင်းသည်) |
| 4 | **Manager Verified (OK)** | Manager verification action | Applicant | No | မန်နေဂျာမှ အတည်ပြုပြီးသည်; လျှောက်ထားသူမှ နောက်ဆုံးအတည်ပြုသူထံ တင်သွင်းရန် စောင့်ဆိုင်းနေသည် |
| 5 | **Rejected by Manager** | Manager rejection action | Applicant | Yes | မန်နေဂျာမှ မှတ်ချက်ဖြင့် ငြင်းပယ်သည်; လျှောက်ထားသူသည် ပြင်ဆင်ပြီး ပြန်လည်တင်သွင်းနိုင်သည် |
| 6 | **Submitted to Approver** | Applicant action (from Manager Verified state only) | Final Approver | No | နောက်ဆုံးအတည်ပြုသူ၏ စစ်ဆေးမှုအတွက် အသင့်ဖြစ်နေသည် |
| 7 | **Approver Reviewing** | Final Approver opens request | Final Approver | No | နောက်ဆုံးအတည်ပြုသူသည် လက်ရှိ စစ်ဆေးနေသည် (နောက်ဆုံးအတည်ပြုသူမှ ကြည့်ရှုသောအခါ အလိုအလျောက်ပြောင်းသည်) |
| 8 | **Approved** | Final Approver approval action | Accounting | No | အတည်ပြုပြီး စာရင်းကိုင်ထံသို့ ငွေပေးချေမှု လုပ်ငန်းစဉ်အတွက် ပေးပို့သည် |
| 9 | **Rejected by Approver** | Final Approver rejection action | Applicant | Yes | နောက်ဆုံးအတည်ပြုသူမှ မှတ်ချက်ဖြင့် ငြင်းပယ်သည်; ပြန်လည်တင်သွင်းပါက လုပ်ငန်းစဉ်ပြန်စရမည် |
| 10 | **Paid (Completed)** | Accounting completion action | N/A | No | ငွေပေးချေမှု လုပ်ငန်းစဉ် ပြီးစီးသည်; လုပ်ငန်းစဉ် ပြီးဆုံးသည် |

### 3.3 အခြေအနေ အပြောင်းအလဲ ဇယား (State Transition Table)

အောက်ပါ အခြေအနေ အပြောင်းအလဲများကိုသာ **ခွင့်ပြုပြီး ကျင့်သုံးသည်**-

```
Draft
  └─> Submitted to Manager (by Applicant)

Submitted to Manager
  └─> Manager Reviewing (automatic, when Manager opens)

Manager Reviewing
  ├─> Manager Verified (OK) (by Manager)
  └─> Rejected by Manager (by Manager)

Manager Verified (OK)
  └─> Submitted to Approver (by Applicant)

Rejected by Manager
  └─> Submitted to Manager (by Applicant after editing)

Submitted to Approver
  └─> Approver Reviewing (automatic, when Final Approver opens)

Approver Reviewing
  ├─> Approved (by Final Approver)
  └─> Rejected by Approver (by Final Approver)

Approved
  └─> Paid (Completed) (by Accounting)

Rejected by Approver
  └─> Submitted to Manager (by Applicant after editing; workflow restarts)

Paid (Completed)
  └─> [END STATE - No further transitions]
```

### 3.4 လုပ်ဆောင်ချက်အလိုက် လိုအပ်ချက်များ (Functional Requirements by Feature)

#### 3.4.1 ငွေပေးချေမှု တောင်းဆိုလွှာ ဖန်တီးခြင်းနှင့် တင်သွင်းခြင်း (Payment Request Creation & Submission)
- **REQ-001:** လျှောက်ထားသူသည် ကနဦး အခြေအနေ "Draft" ဖြင့် ငွေပေးချေမှု တောင်းဆိုလွှာအသစ် ဖန်တီးနိုင်သည်။
- **REQ-002:** လျှောက်ထားသူသည် တောင်းဆိုလွှာကို "Draft" အခြေအနေတွင် ရှိနေစဉ် ပြင်ဆင်နိုင်သည်။
- **REQ-002A:** ငွေပေးချေမှု တောင်းဆိုလွှာ ဖောင်သည် အောက်ပါ မဖြစ်မနေလိုအပ်သော အချက်အလက်များကို ရယူရမည်-
  - Application Date (申請日)
  - Employee Number (社員番号) - User profile မှ အလိုအလျောက် ဖြည့်ပေးမည်
  - Employee Name (氏名) - User profile မှ အလိုအလျောက် ဖြည့်ပေးမည်
  - Total Payment Amount (支払金額) - အသေးစိတ် (breakdown items) များမှ အလိုအလျောက် တွက်ချက်ပေးမည်
  - Desired Payment Date (支払希望日)
  - Currency Type (通貨選択) - dropdown: MMK, USD, JPY စသည်ဖြင့်
  - Payment Type (支払タイプ) - အဖွဲ့အစည်းမှ သတ်မှတ်ထားသော dropdown ရွေးချယ်စရာများ
  - Payment Method (支払方法) - dropdown: Bank Transfer, Cash, Check
  - Purpose/Usage (用途)
  - Bank Account/Phone Info (銀行口座・口座名/電話番号) - ငွေပေးချေမည့် နည်းလမ်းသည် Bank Transfer သို့မဟုတ် Cash ဖြစ်ပါက မဖြစ်မနေ လိုအပ်သည်
  - Receipt Present (領収書の有無) - checkbox/radio (Yes/No)
  - Payment Request Content (支払申請内容) - ဖော်ပြချက်အတွက် text area
  - Target Manager (မန်နေဂျာအခန်းကဏ္ဍရှိသော Active Users များထဲမှ ရွေးချယ်နိုင်သည့် Dropdown)
- **REQ-002B:** Payment Breakdown Table (支払内訳) တွင် စာကြောင်း ၁ မှ ၁၅ ကြောင်းအထိ ပါဝင်နိုင်သည်။ စာကြောင်းတစ်ခုစီတွင် အောက်ပါတို့ ပါဝင်ရမည်-
  - Line Number (No) - ၁-၁၅ အထိ အလိုအလျောက် နံပါတ်စဉ်တပ်ပေးမည်
  - Date (日付) - အသုံးစရိတ် ရက်စွဲ
  - Description (内容) - ဝယ်ယူခဲ့သောအရာ/အသုံးစရိတ် ဖော်ပြချက်
  - Amount (金額) - ဤစာကြောင်းအတွက် ပမာဏ
  - Total (合計) - စာကြောင်းများရှိ ပမာဏအားလုံး၏ စုစုပေါင်း (အလိုအလျောက် တွက်ချက်သည်)
- **REQ-003:** လျှောက်ထားသူသည် ပြေစာဖိုင်များ (ဥပမာ- PDF, JPG, PNG) ကို ငွေပေးချေမှု တောင်းဆိုလွှာတွင် ပူးတွဲတင်နိုင်သည်။ ပူးတွဲဖိုင်ကို Abstracted storage interface ဖြင့် `wwwroot/uploads/` directory တည်ဆောက်ပုံတွင် သိမ်းဆည်းသည်။
- **REQ-003A:** ပြေစာဖိုင် အမည်ပေးခြင်းစနစ်သည် Payment Breakdown အကြောင်းအရာပုံစံ (支払内訳の内容に従って設定) ကို လိုက်နာရမည်။
- **REQ-004:** လျှောက်ထားသူသည် "Draft" တောင်းဆိုလွှာကို မန်နေဂျာထံ တင်သွင်းနိုင်ပြီး၊ status ကို "Submitted to Manager" သို့ ပြောင်းလဲမည်။
- **REQ-005:** လျှောက်ထားသူသည် "Rejected by Manager" သို့မဟုတ် "Rejected by Approver" အခြေအနေရှိ တောင်းဆိုလွှာများကို ပြင်ဆင်နိုင်ပြီး ပြန်လည်တင်သွင်းနိုင်သည်။
- **REQ-006:** တောင်းဆိုလွှာသည် "Manager Verified (OK)" ဖြစ်သွားသည်နှင့်တပြိုင်နက်၊ "Submitted to Approver" အခြေအနေသို့ ကူးပြောင်းရန်အတွက် လျှောက်ထားသူသည် "Submit to Final Approver" ခလုတ်ကို မဖြစ်မနေ နှိပ်ရမည်။

#### 3.4.2 မန်နေဂျာ စိစစ်အတည်ပြုခြင်း လုပ်ငန်းစဉ် (Manager Verification Workflow)
- **REQ-007:** မန်နေဂျာတစ်ဦးသည် "Submitted to Manager" အခြေအနေရှိ တောင်းဆိုလွှာကို ဖွင့်ကြည့်သောအခါ၊ status သည် "Manager Reviewing" သို့ အလိုအလျောက် ပြောင်းလဲသွားမည်။
- **REQ-008:** မန်နေဂျာသည် ပြေစာပူးတွဲဖိုင်များ အပါအဝင် ငွေပေးချေမှု တောင်းဆိုလွှာ၏ အသေးစိတ်အချက်အလက်အားလုံးကို စစ်ဆေးနိုင်သည်။
- **REQ-009:** မန်နေဂျာသည် တောင်းဆိုလွှာကို စိစစ်အတည်ပြု (approve) နိုင်ပြီး၊ status ကို "Manager Verified (OK)" သို့ ပြောင်းလဲမည်။ နောက်ဆုံးအတည်ပြုသူထံ ဆက်လက်တင်ပြနိုင်ရန်အတွက် တောင်းဆိုလွှာသည် လျှောက်ထားသူထံ ပြန်လည်ရောက်ရှိသွားမည်။
- **REQ-010:** မန်နေဂျာသည် တောင်းဆိုလွှာကို ငြင်းပယ်နိုင်ပြီး၊ status ကို "Rejected by Manager" သို့ ပြောင်းလဲမည်။ မဖြစ်မနေ လိုအပ်သော မှတ်ချက် (comment) အကွက်ကို ဖြည့်သွင်းရမည်။
- **REQ-011:** မန်နေဂျာမှ လုပ်ဆောင်ချက်တစ်ခု (စိစစ်ခြင်း သို့မဟုတ် ငြင်းပယ်ခြင်း) ကို ပြုလုပ်သောအခါ၊ ၎င်းလုပ်ဆောင်ချက်နှင့် မှတ်ချက်ကို မှတ်တမ်းတင်ထားသော ApprovalLog တစ်ခုကို ဖန်တီးမည်။

#### 3.4.3 နောက်ဆုံးအတည်ပြုသူ လုပ်ငန်းစဉ် (Final Approver Workflow)
- **REQ-012:** နောက်ဆုံးအတည်ပြုသူသည် "Submitted to Approver" အခြေအနေရှိ တောင်းဆိုလွှာကို ဖွင့်ကြည့်သောအခါ၊ status သည် "Approver Reviewing" သို့ အလိုအလျောက် ပြောင်းလဲသွားမည်။
- **REQ-013:** နောက်ဆုံးအတည်ပြုသူသည် အသေးစိတ်အချက်အလက်များနှင့် အတည်ပြုချက်မှတ်တမ်း (ApprovalLog) အားလုံးကို စစ်ဆေးနိုင်သည်။
- **REQ-014:** နောက်ဆုံးအတည်ပြုသူသည် တောင်းဆိုလွှာကို အတည်ပြုနိုင်ပြီး၊ status ကို "Approved" သို့ ပြောင်းလဲကာ စာရင်းကိုင် စောင့်ဆိုင်းစာရင်းသို့ ပေးပို့မည်။
- **REQ-015:** နောက်ဆုံးအတည်ပြုသူသည် တောင်းဆိုလွှာကို ငြင်းပယ်နိုင်ပြီး၊ status ကို "Rejected by Approver" သို့ ပြောင်းလဲမည်။ မဖြစ်မနေ လိုအပ်သော မှတ်ချက်ကို ဖြည့်သွင်းရမည်။ တောင်းဆိုလွှာသည် လျှောက်ထားသူထံ ပြန်ရောက်သွားမည်။
- **REQ-016:** နောက်ဆုံးအတည်ပြုသူမှ လုပ်ဆောင်ချက်တစ်ခုကို ပြုလုပ်သောအခါ၊ ApprovalLog တစ်ခုကို ဖန်တီးမည်။

#### 3.4.4 စာရင်းကိုင် ငွေပေးချေမှု လုပ်ငန်းစဉ် (Accounting Payment Processing)
- **REQ-017:** စာရင်းကိုင်အဖွဲ့သည် "Approved" အခြေအနေရှိ တောင်းဆိုလွှာအားလုံးကို ပြသထားသော dashboard တစ်ခုကို မြင်တွေ့ရမည်။
- **REQ-018:** စာရင်းကိုင်သည် ပြေစာပူးတွဲဖိုင် အပါအဝင် အတည်ပြုပြီးသော ငွေပေးချေမှု တောင်းဆိုလွှာတစ်ခုစီအတွက် အသေးစိတ်အချက်အလက်များကို ကြည့်ရှုနိုင်သည်။
- **REQ-019:** စာရင်းကိုင်သည် ငွေပေးချေမှုကို "Paid (Completed)" အဖြစ် မှတ်သားနိုင်ပြီး၊ ထိုတောင်းဆိုလွှာအတွက် လုပ်ငန်းစဉ် ပြီးဆုံးမည်။
- **REQ-020:** လျှောက်ထားသူ၏ ဌာနခွဲသည် "Mandalay" ဖြစ်သော တောင်းဆိုလွှာများအတွက်၊ စာရင်းကိုင် dashboard တွင် သိသာထင်ရှားသော သတိပေးချက်တစ်ခုကို ပြသမည်: "Coordinate with Toe San for Cash Payment" (လက်ငင်းငွေချေရန် ဦးတိုးစံနှင့် ညှိနှိုင်းပါ)။
- **REQ-021:** လျှောက်ထားသူ၏ ဌာနခွဲသည် "Mandalay" မဟုတ်သော တောင်းဆိုလွှာများအားလုံးအတွက်၊ စာရင်းကိုင် dashboard တွင် "Standard Bank Transfer" (ပုံမှန်ဘဏ်လွှဲ) ဟု ပြသမည်။
- **REQ-022:** စာရင်းကိုင်မှ တောင်းဆိုလွှာကို paid (ငွေချေပြီး) အဖြစ် မှတ်သားသောအခါ၊ ApprovalLog တစ်ခုကို ဖန်တီးပြီး အချိန်မှတ်တမ်းကို မှတ်သားမည်။

#### 3.4.5 Dashboard နှင့် အခြေအနေ ခြေရာခံခြင်း (Dashboard & Status Tracking)
- **REQ-023:** Applicant dashboard သည် ၎င်းတို့၏ ငွေပေးချေမှု တောင်းဆိုလွှာအားလုံးကို လက်ရှိ status နှင့်အတူ ပြသမည်။
- **REQ-024:** လျှောက်ထားသူသည် မန်နေဂျာနှင့် နောက်ဆုံးအတည်ပြုသူများ၏ ငြင်းပယ်မှုများမှ မှတ်ချက်များနှင့် တုံ့ပြန်ချက်များကို ကြည့်ရှုနိုင်သည်။
- **REQ-025:** Manager dashboard သည် ၎င်းတို့ထံ သတ်မှတ်ပေးထားသော တောင်းဆိုလွှာများကို status အလိုက် အုပ်စုဖွဲ့၍ ပြသမည်။
- **REQ-026:** Final Approver dashboard သည် ၎င်းတို့၏ အတည်ပြုချက်ကို စောင့်ဆိုင်းနေသော တောင်းဆိုလွှာများကို ပြသမည်။
- **REQ-027:** အသုံးပြုသူအားလုံးသည် WebSocket push notifications များမှတစ်ဆင့် အချိန်နှင့်တပြေးညီ status အပ်ဒိတ်များကို ကြည့်ရှုနိုင်သည်။
- **REQ-028:** တောင်းဆိုလွှာတစ်ခုစီသည် ApprovalLog မှတ်တမ်းများနှင့် status အပြောင်းအလဲအားလုံးကို ပြသသော timeline/history မြင်ကွင်းတစ်ခုကို ပြသမည်။

#### 3.4.6 ပြေစာဖိုင် စီမံခန့်ခွဲမှု (Receipt File Management)
- **REQ-029:** လျှောက်ထားသူသည် ပြေစာဖိုင်များ (PDF, JPG, PNG, စသည်ဖြင့်) ကို ဖိုင်အရွယ်အစား ကန့်သတ်ချက် (ဥပမာ- ဖိုင်တစ်ခုလျှင် 10 MB) ဖြင့် တင်နိုင်သည်။
- **REQ-029A:** ငွေပေးချေမှု လျှောက်ထားသည့်အခါ ပြေစာဖိုင် တင်သွင်းခြင်းသည် **အခြေခံအားဖြင့် မဖြစ်မနေ လိုအပ်သည်**။ မန်နေဂျာထံ မတင်သွင်းမီ ဖိုင်ပူးတွဲထားခြင်း ရှိ/မရှိကို စနစ်မှ စစ်ဆေးရမည်။
- **REQ-030:** ပြေစာဖိုင်များကို `wwwroot/uploads/` တွင် တောင်းဆိုလွှာ ID ဖြင့် စီစဉ်ထားသော directory တည်ဆောက်ပုံဖြင့် သိမ်းဆည်းသည် (ဥပမာ- `/uploads/{PaymentRequestID}/{UUID}_{filename}`)။
- **REQ-030A:** ပြေစာဖိုင်အမည်များသည် Payment Breakdown (支払内訳) အကြောင်းအရာပုံစံကို လိုက်နာရမည်။ အကြံပြုထားသော ပုံစံ: `{PaymentBreakdownDescription}_{Date}_{SequenceNumber}.{ext}` (ဥပမာ- "OfficeSupplies_20260609_01.pdf")။
- **REQ-031:** အနာဂတ်တွင် cloud storage (AWS S3, Azure Blob) သို့ ပြောင်းရွှေ့ရာတွင် အထောက်အကူဖြစ်စေရန် ဖိုင်သိမ်းဆည်းခြင်းကို interface တစ်ခု၏ နောက်ကွယ်တွင် ဖုံးကွယ် (abstract) ထားသည်။
- **REQ-032:** ခွင့်ပြုချက်ရရှိထားသော အသုံးပြုသူများ (လျှောက်ထားသူ၊ မန်နေဂျာ၊ အတည်ပြုသူ၊ စာရင်းကိုင်) သာလျှင် ၎င်းတို့ ဝင်ရောက်ကြည့်ရှုခွင့်ရှိသော တောင်းဆိုလွှာများအတွက် ပြေစာဖိုင်များကို ဒေါင်းလုဒ်ဆွဲရန်/ကြည့်ရှုရန် လုပ်ဆောင်နိုင်သည်။
- **REQ-032A:** ငွေပေးချေမှုပြီးစီးပြီးနောက် သို့မဟုတ် တောင်းဆိုလွှာ တင်သွင်းပြီးနောက် စက္ကူပြေစာများ (紙の領収書) ကို စာရင်းကိုင်ထံ တင်သွင်းနိုင်သည်။ စနစ်သည် ဒစ်ဂျစ်တယ်နှင့် ရုပ်ပိုင်းဆိုင်ရာ ပြေစာ နှစ်မျိုးလုံးကို ခြေရာခံသည်။

#### 3.4.8 ရှာဖွေခြင်းနှင့် စစ်ထုတ်ခြင်း (Search & Filtering)
- **REQ-033:** အသုံးပြုသူများသည် တောင်းဆိုလွှာနံပါတ်၊ လျှောက်ထားသူအမည်၊ သို့မဟုတ် ပမာဏအကွာအဝေးတို့ဖြင့် ငွေပေးချေမှု တောင်းဆိုလွှာများကို ရှာဖွေနိုင်သည်။
- **REQ-034:** အသုံးပြုသူများသည် တောင်းဆိုလွှာများကို status၊ ရက်စွဲအကွာအဝေး၊ နှင့် ဌာနခွဲတို့ဖြင့် စစ်ထုတ် (filter) နိုင်သည်။
- **REQ-035:** ရလဒ်များကို စာမျက်နှာခွဲ၍ (paginated) ဖန်တီးသည့်ရက်စွဲ သို့မဟုတ် status ဦးစားပေးမှုအလိုက် စီစဉ်ပြသသည်။

## 4. အထူး လုပ်ငန်းစည်းမျဉ်းများ (Special Business Rules)

### 4.1 လုပ်ငန်းစဉ်ဆိုင်ရာ စည်းမျဉ်းများ (Workflow Rules)

#### စည်းမျဉ်း 4.1.1: မန်နေဂျာသည် စိစစ်သူသာဖြစ်ပြီး ရှေ့ဆက်တင်သွင်းသူမဟုတ်ပါ (Manager is a Verifier, Not a Submitter)
- မန်နေဂျာသည် တောင်းဆိုလွှာများကို စစ်ဆေးပြီး စိစစ်အတည်ပြုသော်လည်း ၎င်းတို့ကို ရှေ့ဆက် **မတင်သွင်းပါ**။
- မန်နေဂျာမှ တောင်းဆိုလွှာကို စိစစ်အတည်ပြု (approve) သောအခါ၊ status သည် "Manager Verified (OK)" ဖြစ်သွားပြီး တောင်းဆိုလွှာသည် လျှောက်ထားသူထံ ပြန်လည်ရောက်ရှိသွားသည်။
- လျှောက်ထားသူသည် လုပ်ငန်းစဉ်ကို ဆက်လက်လုပ်ဆောင်ရန်အတွက် "Submit to Final Approver" ကို ကိုယ်တိုင် နှိပ်ရမည်။
- ၎င်းသည် လျှောက်ထားသူ၏ တာဝန်ယူမှုကို သေချာစေပြီး ဒုတိယအဆင့် ဆုံးဖြတ်ချက်ချမှတ်ရန် နေရာတစ်ခု ထောက်ပံ့ပေးသည်။

#### စည်းမျဉ်း 4.1.2: အသုံးပြုသူ ဝင်ရောက်ကြည့်ရှုချိန်တွင် အလိုအလျောက် Status ပြောင်းလဲခြင်း (Automatic Status Transition on User Access)
- မန်နေဂျာမှ "Submitted to Manager" အခြေအနေရှိ တောင်းဆိုလွှာကို ဖွင့်ကြည့်သောအခါ၊ စနစ်သည် status ကို "Manager Reviewing" သို့ **အလိုအလျောက်** ပြောင်းလဲပေးသည်။
- နောက်ဆုံးအတည်ပြုသူမှ "Submitted to Approver" အခြေအနေရှိ တောင်းဆိုလွှာကို ဖွင့်ကြည့်သောအခါ၊ စနစ်သည် status ကို "Approver Reviewing" သို့ **အလိုအလျောက်** ပြောင်းလဲပေးသည်။
- ၎င်းသည် တောင်းဆိုလွှာကို မည်သူက လက်ရှိ စစ်ဆေးနေကြောင်း အချိန်နှင့်တပြေးညီ ညွှန်ပြချက် ပေးသည်။

#### စည်းမျဉ်း 4.1.3: ငြင်းပယ်ခြင်း (Rejection) အတွက် မဖြစ်မနေ အကြောင်းပြချက်ရေးရမည် (Rejection Requires Mandatory Comment)
- မန်နေဂျာမှ တောင်းဆိုလွှာကို ငြင်းပယ်သောအခါ၊ comment အကွက်သည် **မဖြစ်မနေလိုအပ်ပြီး** အနည်းဆုံး စာလုံးရေ ၁၀ လုံး ပါဝင်ရမည်။
- နောက်ဆုံးအတည်ပြုသူမှ တောင်းဆိုလွှာကို ငြင်းပယ်သောအခါ၊ comment အကွက်သည် **မဖြစ်မနေလိုအပ်ပြီး** အနည်းဆုံး စာလုံးရေ ၁၀ လုံး ပါဝင်ရမည်။
- မှတ်ချက်များကို စစ်ဆေးရေးနှင့် ဆက်သွယ်ရေး ရည်ရွယ်ချက်များအတွက် ApprovalLog တွင် မှတ်တမ်းတင်ထားသည်။

#### စည်းမျဉ်း 4.1.4: ငြင်းပယ်ခံရသော တောင်းဆိုလွှာအတွက် လုပ်ငန်းစဉ် ပြန်စခြင်း (Rejected Request Workflow Restart)
- အကယ်၍ နောက်ဆုံးအတည်ပြုသူမှ တောင်းဆိုလွှာကို ငြင်းပယ်ပါက (status = "Rejected by Approver")၊ လုပ်ငန်းစဉ်သည် မန်နေဂျာမှ **ပြန်လည်စတင်ရမည်**။
- လျှောက်ထားသူသည် တောင်းဆိုလွှာကို ပြင်ဆင်ပြီး "Submitted to Manager" အခြေအနေသစ်ဖြင့် မန်နေဂျာထံ ပြန်လည်တင်သွင်းသည်။
- အတည်ပြုခြင်း အဆင့်ဆင့်တစ်ခုလုံးကို ပြန်လည်လုပ်ဆောင်ရမည်။

#### စည်းမျဉ်း 4.1.5: လျှောက်ထားသူ ပြင်ဆင်နိုင်မှု ကန့်သတ်ချက်များ (Applicant Edit Restrictions)
- လျှောက်ထားသူသည် အောက်ပါ အခြေအနေများရှိ တောင်းဆိုလွှာများကိုသာ ပြင်ဆင်နိုင်သည်-
  - **Draft** (အတည်ပြုချက် မရှိသေးပါ)
  - **Rejected by Manager** (ပြင်ဆင်ပြီး မန်နေဂျာထံ ပြန်လည်တင်သွင်းနိုင်သည်)
  - **Rejected by Approver** (ပြင်ဆင်ပြီး မန်နေဂျာထံ ပြန်လည်တင်သွင်းရမည်)
- တောင်းဆိုလွှာများသည် "Manager Reviewing" သို့မဟုတ် "Approver Reviewing" အခြေအနေများသို့ ရောက်ရှိသွားသည်နှင့် **ပြင်ဆင်၍မရပါ**။
- တောင်းဆိုလွှာတစ်ခုသည် "Approved" ဖြစ်သွားသည်နှင့် လော့ခ်ချခံရပြီး ပြင်ဆင်၍မရပါ (စာရင်းစစ်မှတ်တမ်း တိကျခိုင်မာမှုအတွက်)။

#### စည်းမျဉ်း 4.1.6: မတင်သွင်းမီ ပြေစာဖိုင် မဖြစ်မနေ လိုအပ်သည် (Receipt File is Mandatory Before Submission)
- လျှောက်ထားသူသည် တောင်းဆိုလွှာတစ်ခုကို "Draft" မှ "Submitted to Manager" သို့ မတင်သွင်းမီ အနည်းဆုံး ပြေစာဖိုင်တစ်ခုကို **မဖြစ်မနေ** ပူးတွဲရမည်။
- တင်သွင်းသည့်အခါ ဖိုင်ပူးတွဲထားခြင်းကို စနစ်မှ စစ်ဆေးသည်။

#### စည်းမျဉ်း 4.1.7: ပြေစာ တင်သွင်းခြင်း လိုအပ်ချက်များ (Receipt Submission Requirements)
- **ဒစ်ဂျစ်တယ် ပြေစာ (Digital Receipt File):** ငွေပေးချေမှု တောင်းဆိုလွှာ လျှောက်ထားစဉ်အတွင်း ဒစ်ဂျစ်တယ် ပူးတွဲဖိုင်များအဖြစ် တင်သွင်းသော ပြေစာဖိုင်များ။ ဤဖိုင်များသည် Payment Breakdown အကြောင်းအရာ (支払内訳) ကို အခြေခံ၍ အမည်ပေးခြင်းစနစ်ကို လိုက်နာသည်။
- **စက္ကူပြေစာ (紙の領収書):** ရုပ်ပိုင်းဆိုင်ရာ စက္ကူပြေစာများကို ငွေပေးချေမှု လုပ်ငန်းစဉ် ပြီးစီးပြီးနောက် သို့မဟုတ် တောင်းဆိုလွှာ တင်သွင်းပြီးပြီးချင်း စာရင်းကိုင်ထံ တင်သွင်းနိုင်သည်။ စာရင်းကိုင်သည် ၎င်းတို့ကို သီးခြားသိမ်းဆည်းပြီး ငွေပေးချေမှု တောင်းဆိုလွှာ မှတ်တမ်းနှင့် ချိတ်ဆက်ထားသည်။
- **လက်ဖြင့်ရေးသော ပြေစာများအတွက် ချွင်းချက် (Exception for Manual Receipts):** ပြေစာများကို ချက်ချင်းမရနိုင်သော အခြေအနေများတွင်၊ လျှောက်ထားသူသည် ပြေစာကို နောက်မှပေးမည်ဟု ညွှန်ပြနိုင်သော်လည်း လိုအပ်ပါက အတည်မပြုမီ မန်နေဂျာ/နောက်ဆုံးအတည်ပြုသူမှ ဒစ်ဂျစ်တယ်မိတ္တူကို တောင်းဆိုနိုင်သည်။

### 4.2 အသေးစိတ်ရှင်းတမ်း လုပ်ငန်းစည်းမျဉ်းများ (Payment Breakdown Business Rules)

#### စည်းမျဉ်း 4.2.1: အသေးစိတ်ရှင်းတမ်း တွက်ချက်မှု (Payment Breakdown Calculation)
- Total Amount (支払金額) သည် Payment Breakdown ရှိ item ပမာဏအားလုံး၏ စုစုပေါင်း (支払内訳 合計) နှင့် **ညီရမည်**။
- စနစ်သည် အသေးစိတ် (breakdown items) များမှ စုစုပေါင်းကို အလိုအလျောက် တွက်ချက်ပေးသည်; အသုံးပြုသူများသည် စုစုပေါင်းပမာဏကို ကိုယ်တိုင် ထည့်သွင်း၍မရပါ။
- breakdown ရှိ စာကြောင်းတစ်ခုစီသည် နောက်ဆုံးပမာဏတွင် ပါဝင်တွက်ချက်သည်။
- စာကြောင်းတစ်ခုကို ဖျက်လိုက်ပါက၊ စုစုပေါင်းကို အလိုအလျောက် ပြန်လည်တွက်ချက်သည်။

#### စည်းမျဉ်း 4.2.2: အသေးစိတ်ရှင်းတမ်း စာကြောင်းများ (Payment Breakdown Line Items)
- Breakdown items များသည် စာကြောင်း ၁၅ ကြောင်းအထိ ပံ့ပိုးပေးသည် (No: 1-15)။
- စာကြောင်းတိုင်းတွင် Date (日付)၊ Description (内容) နှင့် Amount (金額) တို့ ပါဝင်ရမည်။
- စာကြောင်းများကို အစီအစဉ်အတိုင်း ပြသသည်; အသုံးပြုသူများသည် တည်းဖြတ်သည့် ဝင်းဒိုးအတွင်း (Draft အခြေအနေတွင်သာ) စာကြောင်းများကို ကြားညှပ်ခြင်း၊ ပြင်ဆင်ခြင်း သို့မဟုတ် ဖျက်ခြင်းများ ပြုလုပ်နိုင်သည်။
- တောင်းဆိုလွှာတစ်ခုသည် "Draft" အခြေအနေမှ ကျော်လွန်သွားသည်နှင့်၊ breakdown items များကို ပြင်ဆင်၍မရတော့သော်လည်း စစ်ဆေးသူအားလုံး ကြည့်ရှုနိုင်သည်။

#### စည်းမျဉ်း 4.2.3: အသေးစိတ်ရှင်းတမ်းနှင့် ပြေစာဖိုင် အမည်ပေးခြင်း (Payment Breakdown and Receipt File Naming)
- ပြေစာဖိုင် အမည်များသည် Payment Breakdown အကြောင်းအရာကို လိုက်နာသင့်သည် (支払内訳の内容に従って設定)။
- အကြံပြုထားသော ပုံစံ: breakdown စာကြောင်း၏ ဖော်ပြချက်သည် "Office Supplies" ဖြစ်ပါက၊ ပြေစာအမည်ကို "OfficeSupplies_20260609.pdf" ကဲ့သို့ ပေးသင့်သည်။
- စနစ်သည် လျှောက်ထားသူများအား ၎င်းတို့ထည့်သွင်းထားသော breakdown ဖော်ပြချက်များကို ပြသပေးသည့် ဖိုင်မှတ်တမ်း လမ်းညွှန် (filing guide) တစ်ခုကို ထောက်ပံ့ပေးနိုင်ပြီး၊ ၎င်းတို့၏ ပြေစာဖိုင်များကို သင့်လျော်စွာ အမည်ပေးနိုင်ရန် ကူညီပေးသည်။

### 4.3 ဌာနခွဲအလိုက် လုပ်ငန်းစည်းမျဉ်းများ (Branch-Specific Business Rules)

#### စည်းမျဉ်း 4.3.1: Mandalay ဌာနခွဲ လက်ငင်းငွေချေမှု သတိပေးချက် (Mandalay Branch Cash Payment Alert)
- အကယ်၍ လျှောက်ထားသူ၏ ဌာနခွဲသည် **"Mandalay"** ဖြစ်ပါက၊ စာရင်းကိုင်အဖွဲ့သည် အောက်ပါ သိသာထင်ရှားသော သတိပေးချက်ကို မြင်တွေ့ရမည်:
  - **"IMPORTANT: Coordinate with Toe San for Cash Payment"**
- ဤသတိပေးချက်သည် အတည်ပြုပြီးသော Mandalay တောင်းဆိုလွှာတိုင်းအတွက် စာရင်းကိုင် dashboard တွင် ပေါ်လာမည်။
- သတိပြုမိစေရန် သတိပေးချက်ကို အရောင်ပြောင်းထားသည် (ဥပမာ- အနီရောင်/သတိပေးအရောင်)။

#### စည်းမျဉ်း 4.3.2: Mandalay မဟုတ်သော ပုံမှန်လုပ်ငန်းစဉ် (Non-Mandalay Standard Processing)
- **Mandalay မှလွဲ၍** အခြားဌာနခွဲအားလုံးအတွက်၊ စာရင်းကိုင်သည် အောက်ပါကို မြင်တွေ့ရမည်:
  - **"Standard Bank Transfer Processing"**
- ၎င်းသည် ပုံမှန် ဘဏ်လွှဲလုပ်ငန်းစဉ်ကို အသုံးပြုကြောင်း ညွှန်ပြသည်။

### 4.4 ဒေတာ တိကျခိုင်မာမှုနှင့် စစ်ဆေးရေး စည်းမျဉ်းများ (Data Integrity & Audit Rules)

#### စည်းမျဉ်း 4.4.1: မပြင်ဆင်နိုင်သော အတည်ပြုချက် မှတ်တမ်း (Immutable Approval Log)
- အခြေအနေ အပြောင်းအလဲ၊ လုပ်ဆောင်ချက်နှင့် မှတ်ချက်တိုင်းကို ApprovalLog တွင် မှတ်တမ်းတင်ထားပြီး **ဖျက်ခြင်း သို့မဟုတ် ပြင်ဆင်ခြင်း မပြုလုပ်နိုင်ပါ**။
- ApprovalLog သည် စည်းမျဉ်းလိုက်နာမှုနှင့် အငြင်းပွားမှု ဖြေရှင်းခြင်းများအတွက် အမှန်တရား၏ ရင်းမြစ် (source of truth) အဖြစ် ဆောင်ရွက်သည်။

#### စည်းမျဉ်း 4.4.2: တောင်းဆိုလွှာများအတွက် Soft Delete အသုံးပြုခြင်း (Soft Delete for Requests)
- ငွေပေးချေမှု တောင်းဆိုလွှာများကို database မှ **မည်သည့်အခါမျှ အပြီးတိုင် မဖျက်ပါ**။
- ၎င်းအစား မှတ်တမ်းအဖြစ်သိမ်းဆည်းသော သို့မဟုတ် ပယ်ဖျက်သော တောင်းဆိုလွှာများအတွက် "IsDeleted" flag ကို true အဖြစ် သတ်မှတ်သည်။
- ဖျက်လိုက်သော တောင်းဆိုလွှာများကို ပုံမှန် dashboard များမှ ဖျောက်ထားသော်လည်း စစ်ဆေးရေး ရည်ရွယ်ချက်များအတွက် သိမ်းဆည်းထားသည်။
- **Draft ဖျက်သိမ်းခွင့် အထူးစည်းမျဉ်း (Draft Deletion Privilege)**: လျှောက်ထားသူများသည် မိမိတို့၏ ငွေပေးချေမှုတောင်းဆိုလွှာများကို Draft အခြေအနေ (DRAFT status) တွင် ရှိနေစဉ်သာ Soft-Delete ပြုလုပ်ခွင့်ရှိသည်။ အခြား မည်သည့် status တွင်မဆို ဖျက်သိမ်းခွင့်ကို ပိတ်ပင်ထားပြီး ခလုတ်ကို ဝှက်ထားရမည်ဖြစ်ကာ တောင်းဆိုလွှာသည် Read-only အဖြစ်သာ တည်ရှိရမည်။

#### စည်းမျဉ်း 4.4.3: အချိန်မှတ်တမ်း ခြေရာခံခြင်း (Timestamp Tracking)
- အခြေအနေ အပြောင်းအလဲ၊ လုပ်ဆောင်ချက်များနှင့် ဝင်ရောက်ကြည့်ရှုမှု အဖြစ်အပျက်များ အားလုံးကို အချိန်မှတ်တမ်း (timestamp) ဖြင့် မှတ်သားထားသည်။
- အချိန်ဇုန် (time zone) များတစ်လျှောက် တစ်သမတ်တည်းဖြစ်စေရန် အချိန်မှတ်တမ်းများကို UTC ပုံစံဖြင့် မှတ်တမ်းတင်ထားသည်။

## 5. လုပ်ဆောင်ချက်မဟုတ်သော လိုအပ်ချက်များ (Non-Functional Requirements)

### 5.1 စွမ်းဆောင်ရည် (Performance)

- **NFR-001:** Dashboard များအတွက် စာမျက်နှာ ဖွင့်ချိန်သည် **≤ ၂ စက္ကန့်** ဖြစ်သင့်သည် (ကနဦး တောင်းဆိုမှုမှ အပြည့်အဝ render လုပ်ချိန်အထိ တိုင်းတာသည်)။
- **NFR-002:** ရှာဖွေခြင်းနှင့် စစ်ထုတ်ခြင်း (filter) လုပ်ဆောင်ချက်များသည် မှတ်တမ်းပေါင်း ၁၀,၀၀၀ အထိရှိသော data များအတွက် ရလဒ်များကို **≤ ၃ စက္ကန့်** အတွင်း ပြန်ပေးသင့်သည်။
- **NFR-003:** SignalR မှတစ်ဆင့် အချိန်နှင့်တပြေးညီ အခြေအနေ အပ်ဒိတ်များကို အခြေအနေ ပြောင်းလဲပြီးနောက် **≤ ၅၀၀ မီလီစက္ကန့်** အတွင်း ပေးပို့သင့်သည်။
- **NFR-004:** မကြာခဏ ရှာဖွေလေ့ရှိသော အကွက်များ (Status, ApplicantUserID, ManagerUserID, FinalApproverUserID, CreatedDate) အတွက် Database query များသည် သင့်လျော်သော indexing ကို အသုံးပြုသင့်သည်။

### 5.2 လုံခြုံရေးနှင့် ဝင်ရောက်ခွင့် ထိန်းချုပ်မှု (Security & Access Control)

- **NFR-005:** API endpoint များနှင့် page handler များအားလုံးသည် ကဏ္ဍအလိုက် (role-based) ခွင့်ပြုချက် စစ်ဆေးမှုများကို အကောင်အထည်ဖော်ရမည်။
- **NFR-006:** အသုံးပြုသူအားလုံးအတွက် အထောက်အထား စစ်ဆေးခြင်း (Authentication) လိုအပ်သည်; ASP.NET Core Identity သို့မဟုတ် ၎င်းနှင့် ညီမျှသောအရာမှတစ်ဆင့် session-based authentication ကို အကောင်အထည်ဖော်ပါ။
- **NFR-007:** အရေးကြီးသော လုပ်ဆောင်ချက်များ (အတည်ပြုခြင်း၊ ငြင်းပယ်ခြင်း၊ ငွေချေပြီးအဖြစ် မှတ်သားခြင်း) သည် အသုံးပြုသူ အထောက်အထား စစ်ဆေးအတည်ပြုရန် လိုအပ်သည်။
- **NFR-008:** ဖိုင်ဒေါင်းလုဒ်များနှင့် ငွေပေးချေမှု တောင်းဆိုလွှာ အသေးစိတ်များကို ဝင်ရောက်ကြည့်ရှုခြင်းသည် အသုံးပြုသူ၏ ခွင့်ပြုချက် (ကဏ္ဍနှင့် တောင်းဆိုလွှာ သတ်မှတ်ချက်) ကို စစ်ဆေးရမည်။
- **NFR-009:** Parameterized query များ (Entity Framework Core) မှတစ်ဆင့် SQL injection ကို ကာကွယ်ခြင်း; အသုံးပြုသူ ထည့်သွင်းမှုအားလုံးအတွက် input validation ပြုလုပ်ခြင်း။
- **NFR-010:** အခြေအနေ ပြောင်းလဲစေသော လုပ်ဆောင်ချက်အားလုံးတွင် CSRF (Cross-Site Request Forgery) ကာကွယ်မှု (ASP.NET Core anti-forgery tokens များကို အသုံးပြုပါ)။
- **NFR-011:** စည်းမျဉ်းလိုက်နာမှု ခြေရာခံခြင်းအတွက် အရေးကြီးသော လုပ်ဆောင်ချက်အားလုံး၏ IP address များနှင့် user ID များကို Audit log မှတ်တမ်းတင်ပါ။

### 5.3 ဒေတာ သိမ်းဆည်းခြင်းနှင့် ဖိုင်စီမံခန့်ခွဲမှု (Data Storage & File Management)

- **NFR-012:** ပြဿနာများကို ကာကွယ်ရန်အတွက် ပြေစာဖိုင်များကို `/uploads/{PaymentRequestID}/{UUID}_{filename}` တည်ဆောက်ပုံဖြင့် `wwwroot/uploads/` directory တွင် သိမ်းဆည်းထားသည်။
- **NFR-013:** လုပ်ငန်းဆိုင်ရာ ယုတ္တိ (business logic) ကို ကုဒ်ပြောင်းလဲစရာမလိုဘဲ အနာဂတ်တွင် AWS S3, Azure Blob သို့မဟုတ် အခြား cloud storage သို့ ပြောင်းရွှေ့ခွင့်ပြုရန် ဖိုင်သိမ်းဆည်းခြင်း ယုတ္တိကို `IFileStorageService` interface ၏ နောက်ကွယ်တွင် ဖုံးကွယ် (abstract) ထားသည်။
- **NFR-014:** ပြေစာတစ်ခုစီအတွက် အများဆုံး ဖိုင်အရွယ်အစား: **10 MB**။ တောင်းဆိုလွှာတစ်ခုစီအတွက် စုစုပေါင်း ပူးတွဲဖိုင်အရွယ်အစား: **50 MB**။
- **NFR-015:** ပံ့ပိုးထားသော ဖိုင်အမျိုးအစားများ: PDF, JPEG, JPG, PNG (MIME type စစ်ဆေးမှုဖြင့်)။
- **NFR-016:** တောင်းဆိုလွှာများကို soft-delete လုပ်သည့်အခါ ဖိုင်များကို မဖျက်ပါ; ရေရှည်သိမ်းဆည်းရန်အတွက် မှတ်တမ်းတင် သိမ်းဆည်းခြင်း မဟာဗျူဟာ (archival strategy) ကို စဉ်းစားပါ။

### 5.4 အချိန်နှင့်တပြေးညီ ဆက်သွယ်မှု (Real-Time Communication)

- **NFR-017:** အချိန်နှင့်တပြေးညီ push notification များအတွက် WebSocket (Socket.io) ကို အကောင်အထည်ဖော်ထားသည်။
- **NFR-018:** အခြေအနေ ပြောင်းလဲမှု အသိပေးချက်များကို ချိတ်ဆက်ထားသော client များထံ အချိန်နှင့်တပြေးညီ ပေးပို့သည် (≤ 500 ms)။
- **NFR-019:** အသုံးပြုသူတစ်ဦးစီသည် ၎င်းတို့၏ ကဏ္ဍ (ဥပမာ- "Managers", "Applicants", "Approvers", "Accounting") အပေါ်အခြေခံ၍ WebSocket room တစ်ခုသို့ ချိတ်ဆက်ထားသည်။
- **NFR-020:** လျှောက်ထားသူများသည် ၎င်းတို့၏ တောင်းဆိုလွှာ အခြေအနေ ပြောင်းလဲသွားသည့်အခါ (Manager Reviewing, Manager Verified, Approved, Rejected, Paid) အသိပေးချက်များကို လက်ခံရရှိသည်။
- **NFR-021:** မန်နေဂျာများသည် တောင်းဆိုလွှာအသစ်တစ်ခုကို ၎င်းတို့ထံ သတ်မှတ်ပေးလိုက်သည့်အခါ အသိပေးချက်များကို လက်ခံရရှိသည်။
- **NFR-022:** ကွန်ရက် အနှောင့်အယှက်များအတွက် အဆင်ပြေချောမွေ့စွာ ပြန်လည်ချိတ်ဆက်ခြင်း (graceful reconnection) နှင့် မက်ဆေ့ဂျ်များကို တန်းစီစောင့်ဆိုင်းခြင်း (message queueing) တို့ ပါဝင်သည်။

### 5.5 Caching နှင့် စွမ်းဆောင်ရည် (Redis)

- **NFR-023:** စွမ်းဆောင်ရည်မြှင့်တင်ရန်အတွက် session စီမံခန့်ခွဲမှုနှင့် အလွန်မြန်ဆန်သော caching အတွက် Redis ကို အသုံးပြုသည်။
- **NFR-024:** လုံခြုံရေးအတွက် သတ်မှတ်နိုင်သော TTL (Time-To-Live) ဖြင့် User session များကို Redis တွင် သိမ်းဆည်းထားသည်။
- **NFR-025:** မကြာခဏ ဝင်ရောက်ကြည့်ရှုလေ့ရှိသော lookup ဇယားများ (UserRoles, PaymentStatuses, PaymentTypes, စသည်) ကို Redis တွင် cache လုပ်ထားသည်။
- **NFR-026:** Cache invalidation မဟာဗျူဟာ: ဒေတာ အပ်ဒိတ်လုပ်သည့်အခါ အလိုအလျောက် သက်တမ်းကုန်ဆုံးခြင်းနှင့် ကိုယ်တိုင် invalidation ပြုလုပ်ခြင်း။
- **NFR-027:** Redis connection pooling ကို ဝင်ရောက်အသုံးပြုမှုများသောအချိန်တွင် (concurrent load) အကောင်းဆုံး စွမ်းဆောင်ရည်အတွက် ပြင်ဆင်သတ်မှတ်ထားသည်။

### 5.5 Database နှင့် ORM (Database & ORM)

- **NFR-023:** အဓိက database အဖြစ် Microsoft SQL Server ကို အသုံးပြုသည်။
- **NFR-024:** `Microsoft.EntityFrameworkCore.SqlServer` provider ဖြင့် Entity Framework Core (EF Core)။
- **NFR-025:** Database schema ကို EF Core Migrations မှတစ်ဆင့် ဗားရှင်းထိန်းချုပ်ထားသည်။
- **NFR-026:** Foreign key များနှင့် မကြာခဏ စစ်ထုတ်လေ့ရှိသော ကော်လံများ (Status, UserID, CreatedDate) တွင် သင့်လျော်သော index များ။
- **NFR-027:** Database backup မဟာဗျူဟာကို သတ်မှတ်ထားသည်; နေ့စဉ် အလိုအလျောက် backup များကို အကြံပြုထားသည်။

### 5.6 Deployment နှင့် ပတ်ဝန်းကျင် (Deployment & Environment)

- **NFR-028:** Local development ပတ်ဝန်းကျင်: .NET CLI နှင့် Kestrel server ပါဝင်သော Visual Studio Code (VSCode)။
- **NFR-029:** ပံ့ပိုးထားသော deployment: IIS ပါဝင်သော Windows Server သို့မဟုတ် containerized Docker deployment။
- **NFR-030:** `appsettings.json` နှင့် ပတ်ဝန်းကျင်အလိုက် (development, staging, production) config ဖိုင်များမှတစ်ဆင့် Configuration စီမံခန့်ခွဲမှု။
- **NFR-031:** Database connection string များကို လုံခြုံစွာ သိမ်းဆည်းထားသည် (Azure Key Vault သို့မဟုတ် environment variables)။
- **NFR-032:** Serilog သို့မဟုတ် အလားတူ structured logging framework မှတစ်ဆင့် Application logging။

### 5.7 တိုးချဲ့နိုင်စွမ်းနှင့် ထိန်းသိမ်းခြင်း (Scalability & Maintenance)

- **NFR-033:** ကုဒ်ကို Controllers, Services, Repositories, Models ဟူ၍ ရှင်းလင်းစွာ ခွဲခြား၍ ဖွဲ့စည်းထားသည်။
- **NFR-034:** ထိန်းသိမ်းရလွယ်ကူမှုနှင့် စမ်းသပ်နိုင်မှုအတွက် Business logic ကို service class များသို့ ခွဲထုတ် (abstract) ထားသည်။
- **NFR-035:** ဖိုင်သိမ်းဆည်းခြင်း abstraction သည် ကြီးမားသော refactoring မလိုဘဲ အနာဂတ်တွင် cloud storage သို့ ပြောင်းလဲခြင်းကို ခွင့်ပြုသည်။
- **NFR-036:** Database သည် သိသာထင်ရှားသော စွမ်းဆောင်ရည်ကျဆင်းမှုမရှိဘဲ ငွေပေးချေမှု တောင်းဆိုလွှာ ၁၀၀,၀၀၀+ အထိ တိုးချဲ့နိုင်ရန် ဒီဇိုင်းဆွဲထားသည်။

### 5.8 စည်းမျဉ်းလိုက်နာမှုနှင့် စစ်ဆေးရေး (Compliance & Audit)

- **NFR-037:** အခြေအနေ အပြောင်းအလဲအားလုံးကို အချိန်မှတ်တမ်း၊ user ID၊ လုပ်ဆောင်ချက် အမျိုးအစားနှင့် ရွေးချယ်နိုင်သော မှတ်ချက်များဖြင့် မှတ်တမ်းတင်ထားသည်။
- **NFR-038:** Audit log များသည် ပြင်ဆင်၍မရပါ (immutable) နှင့် အနည်းဆုံး ၅ နှစ် သိမ်းဆည်းထားရမည် (စည်းမျဉ်းဆိုင်ရာ လိုအပ်ချက်)။
- **NFR-039:** စနစ်သည် ငွေပေးချေမှု တောင်းဆိုလွှာ၏ သက်တမ်းတစ်လျှောက်နှင့် ပြုလုပ်ခဲ့သော လုပ်ဆောင်ချက်အားလုံးကို ပြသသည့် audit အစီရင်ခံစာများကို ထုတ်ပေးသည်။
- **NFR-040:** အရေးကြီးသော အကွက်များ (ဥပမာ- တိကျသော ငွေပေးချေမှု ပမာဏများ၊ လျှောက်ထားသူ၏ ကိုယ်ရေးကိုယ်တာ အချက်အလက်) သည် ဒေတာ ကာကွယ်ရေး စည်းမျဉ်းများနှင့် သက်ဆိုင်နိုင်သည်; သင့်လျော်သော ဝင်ရောက်ခွင့် ထိန်းချုပ်မှုများကို အကောင်အထည်ဖော်ပါ။

### 5.9 အသုံးပြုရလွယ်ကူမှု (Usability)

- **NFR-041:** UI သည် responsive ဖြစ်ပြီး desktop၊ tablet နှင့် mobile browser များတွင် အလုပ်လုပ်သည်။
- **NFR-042:** လမ်းညွှန်မှု (Navigation) သည် နားလည်ရလွယ်ကူသည်; ကဏ္ဍတစ်ခုစီတွင် ၎င်းတို့၏ အဓိက လုပ်ငန်းစဉ်အတွက် ရှင်းလင်းသော လမ်းကြောင်းတစ်ခု ရှိသည်။
- **NFR-043:** Error message များသည် ရှင်းလင်းပြီး လက်တွေ့လုပ်ဆောင်နိုင်သည် (ဥပမာ- "မတင်သွင်းမီ ပြေစာဖိုင် လိုအပ်သည်")။
- **NFR-044:** Dashboard များပေါ်ရှိ အချိန်နှင့်တပြေးညီ အခြေအနေ အပ်ဒိတ်များသည် စာမျက်နှာကို refresh လုပ်ရန် မလိုအပ်ပါ။

## 6. စနစ် တည်ဆောက်ပုံ အကြောင်းအရာ (System Architecture Context)

### 6.1 နည်းပညာ ဆုံးဖြတ်ချက် အကြောင်းရင်း (Technology Decision Rationale)

#### ASP.NET Core MVC/Razor Pages
- **အဘယ်ကြောင့်နည်း (Why):** C# ပါဝင်သော ပေါင်းစပ် model-view-controller ပုံစံသည် strong typing, IntelliSense အထောက်အပံ့နှင့် လုံခြုံရေး အင်္ဂါရပ်များ (CSRF ကာကွယ်မှု၊ input validation) ကို ထောက်ပံ့ပေးသည်။
- **Razor Pages** သည် routing ရှုပ်ထွေးမှုမရှိဘဲ ကဏ္ဍအလိုက် dashboard များအတွက် သန့်ရှင်းပြီး အာရုံစိုက်ထားသော စာမျက်နှာများကို ထောက်ပံ့ပေးသည်။
- **MVC Controllers** သည် API endpoint များနှင့် ရှုပ်ထွေးသော workflow ယုတ္တိအတွက် ဖြစ်သည်။

#### Microsoft SQL Server
- **အဘယ်ကြောင့်နည်း (Why):** Workflow နှင့် audit log လိုအပ်ချက်များအတွက် သင့်လျော်သော၊ ခိုင်မာသော ACID compliance ပါဝင်သည့် လုပ်ငန်းသုံး relational database။
- **Transactional integrity** သည် လုပ်ငန်းဆောင်ရွက်နေစဉ်အတွင်း အခြေအနေ အပြောင်းအလဲများ ပျက်စီးမှုမရှိစေရန် သေချာစေသည်။

#### Entity Framework Core
- **အဘယ်ကြောင့်နည်း (Why):** ORM သည် ဒေတာဝင်ရောက်မှုကို ရိုးရှင်းစေပြီး compile-time ဘေးကင်းမှုဖြင့် LINQ query များကို ခွင့်ပြုသည်။
- **Migrations** သည် ကုဒ်နှင့်အတူ database schema ကို ဗားရှင်းထိန်းချုပ်သည်။
- **Abstraction** သည် လိုအပ်ပါက အနာဂတ် database အပြောင်းအလဲများကို ပံ့ပိုးပေးသည်။

#### ASP.NET Core SignalR
- **အဘယ်ကြောင့်နည်း (Why):** အချိန်နှင့်တပြေးညီ နှစ်လမ်းသွား ဆက်သွယ်မှုသည် dashboard များပေါ်တွင် တိုက်ရိုက် အခြေအနေ အပ်ဒိတ်များကို လုပ်ဆောင်နိုင်စေသည်။
- **Group-based messaging** သည် တိကျသော ကဏ္ဍများ/အသုံးပြုသူများကို ထိရောက်စွာ ပစ်မှတ်ထားနိုင်စေသည်။
- **Automatic reconnection** သည် ယာယီ ကွန်ရက်ပြဿနာများကို ကိုင်တွယ်ဖြေရှင်းပေးသည်။

#### Local File Storage ဖြင့် Abstraction (Local File Storage with Abstraction)
- **အဘယ်ကြောင့်နည်း (Why):** `wwwroot/uploads/` သည် local development အတွက် ရိုးရှင်းသော ကနဦး သိမ်းဆည်းမှုကို ထောက်ပံ့ပေးသည်။
- **`IFileStorageService` abstraction** သည် business logic ကို မပြောင်းလဲဘဲ AWS S3, Azure Blob Storage သို့မဟုတ် အခြား cloud provider များသို့ ပြောင်းရွှေ့ခြင်းကို လုပ်ဆောင်နိုင်စေသည်။
- **အနာဂတ်အတွက် ပြင်ဆင်ခြင်း (Future-proofing):** အဖွဲ့အစည်း၏ လိုအပ်ချက်များ ပြောင်းလဲသွားသည့်အခါ refactoring ကို အနည်းဆုံးဖြစ်စေသည်။

### 6.3 ဖွံ့ဖြိုးတိုးတက်မှု ကန့်သတ်ချက်များနှင့် ယူဆချက်များ (Development Constraints & Assumptions)

- **Single-Tenant:** စနစ်ကို အဖွဲ့အစည်းတစ်ခုတည်းအတွက် ဒီဇိုင်းဆွဲထားသည်; multi-tenancy သည် ဘောင်ပြင်ပတွင် ရှိသည်။
- **English Language:** UI နှင့် business logic များကို အင်္ဂလိပ်ဘာသာဖြင့် မှတ်တမ်းတင်ထားသည်; ဤအဆင့်တွင် ဘာသာပြန်ဆိုမှု (localization) မလိုအပ်ပါ။
- **Local Environment:** Development သည် Visual Studio Code (VSCode), .NET CLI, Kestrel server, နှင့် local SQL Server ကို အသုံးပြုသည်ဟု ယူဆသည်။
- **Reasonable User Load:** ပုံမှန် လုပ်ငန်းသုံး workflow အတွက် ဒီဇိုင်းဆွဲထားသည် (တစ်ပြိုင်နက်တည်း အသုံးပြုသူ ထောင်ပေါင်းများစွာမဟုတ်ဘဲ၊ တစ်နေ့လျှင် တောင်းဆိုလွှာ ရာဂဏန်း)။
- **Existing Infrastructure:** အဖွဲ့အစည်းတွင် SQL Server နှင့် Windows Server အခြေခံအဆောက်အအုံ ရရှိနိုင်သည်ဟု ယူဆသည်။

### 6.4 အနာဂတ်တွင် တိုးချဲ့နိုင်မည့် အချက်များ (Future Extensibility Points)

1. **Cloud File Storage:** အခြားရွေးချယ်စရာ `IFileStorageService` ကို အကောင်အထည်ဖော်ခြင်းဖြင့် local file storage ကို AWS S3 သို့မဟုတ် Azure Blob ဖြင့် အစားထိုးရန်။
2. **Email Notifications:** Status အပြောင်းအလဲများတွင် အီးမေးလ် အသိပေးချက်များ ပေးပို့ရန် ApprovalLog ကို တိုးချဲ့ရန် (လက်ရှိ ဘောင်တွင် မပါဝင်ပါ)။
3. **Advanced Reporting:** ApprovalLog audit trail ကို အသုံးပြု၍ အစီရင်ခံစာ ထုတ်လုပ်ခြင်း (PDF exports, analytics dashboards) ထည့်သွင်းရန်။
4. **Workflow Customization:** ဌာနခွဲ သို့မဟုတ် ဌာနအလိုက် အတည်ပြုခြင်း အဆင့်ဆင့်ကို ပြင်ဆင်သတ်မှတ်နိုင်ရန် workflow engine ကို အကောင်အထည်ဖော်ရန်။
5. **Integration with Accounting Systems:** အလိုအလျောက် ငွေပေးချေမှု ပြုလုပ်ရန်အတွက် ERP စနစ်များ (SAP, NetSuite) နှင့် အနာဂတ်တွင် ချိတ်ဆက်ရန်။
6. **Mobile Application:** သွားရင်းလာရင်း အတည်ပြုချက်များ ပြုလုပ်ရန်အတွက် တူညီသော API များကို အသုံးပြုသည့် Native mobile app ထည့်သွင်းရန်။

## 7. လက်ခံနိုင်သော စံနှုန်းများနှင့် အောင်မြင်မှု တိုင်းတာချက်များ (Acceptance Criteria & Success Metrics)

### 7.1 လုပ်ဆောင်ချက်ဆိုင်ရာ လက်ခံနိုင်သော စံနှုန်းများ (Functional Acceptance Criteria)

- [ ] အသုံးပြုသူ ကဏ္ဍ လေးခုစလုံးသည် log in ဝင်နိုင်ပြီး ၎င်းတို့၏ သက်ဆိုင်ရာ dashboard များကို ဝင်ရောက်ကြည့်ရှုနိုင်သည်။
- [ ] ငွေပေးချေမှု တောင်းဆိုလွှာ ဖန်တီးခြင်း၊ တင်သွင်းခြင်းနှင့် သိမ်းဆည်းခြင်းတို့သည် လျှောက်ထားသူအတွက် အစမှအဆုံး အလုပ်လုပ်သည်။
- [ ] မန်နေဂျာသည် အလိုအလျောက် status အပြောင်းအလဲများဖြင့် တောင်းဆိုလွှာများကို စိစစ်အတည်ပြု/ငြင်းပယ်နိုင်သည်။
- [ ] နောက်ဆုံးအတည်ပြုသူသည် တောင်းဆိုလွှာများကို အတည်ပြု/ငြင်းပယ်နိုင်သည်; ငြင်းပယ်ခံရသော တောင်းဆိုလွှာများသည် လုပ်ငန်းစဉ်ပြန်စရန်အတွက် လျှောက်ထားသူထံ ပြန်ရောက်သွားသည်။
- [ ] စာရင်းကိုင်သည် အတည်ပြုပြီးသော စောင့်ဆိုင်းစာရင်းကို ကြည့်ရှုနိုင်သည်၊ ငွေပေးချေမှုများကို ပြီးစီးကြောင်း မှတ်သားနိုင်သည်၊ နှင့် ဌာနခွဲအလိုက် သတိပေးချက်များကို မြင်တွေ့နိုင်သည်။
- [ ] အခြေအနေ အပြောင်းအလဲအားလုံးသည် State Transition Table (အပိုင်း 3.3) ကို အတိအကျ လိုက်နာသည်။
- [ ] ပြေစာဖိုင်များကို တင်နိုင်သည်၊ သိမ်းဆည်းနိုင်သည်၊ နှင့် ခွင့်ပြုချက်ရထားသော အသုံးပြုသူများမှ ပြန်လည်ရယူနိုင်သည်။
- [ ] အချိန်နှင့်တပြေးညီ status အပ်ဒိတ်များသည် SignalR မှတစ်ဆင့် စာမျက်နှာကို refresh လုပ်စရာမလိုဘဲ dashboard များပေါ်တွင် ပေါ်လာသည်။
- [ ] Audit log သည် အချိန်မှတ်တမ်းများ၊ user ID များနှင့် မှတ်ချက်များဖြင့် လုပ်ဆောင်ချက်အားလုံးကို ဖမ်းယူထားသည်။

### 7.2 လုပ်ဆောင်ချက်မဟုတ်သော လက်ခံနိုင်သော စံနှုန်းများ (Non-Functional Acceptance Criteria)

- [ ] Dashboard စာမျက်နှာများသည် ≤ ၂ စက္ကန့် အတွင်း load ဖြစ်သည်။
- [ ] SignalR status အပ်ဒိတ်များသည် ≤ ၅၀၀ မီလီစက္ကန့် အတွင်း ရောက်ရှိသည်။
- [ ] ကဏ္ဍအလိုက် (role-based) ဝင်ရောက်ခွင့် ထိန်းချုပ်မှုအားလုံးကို ကျင့်သုံးထားသည်; ခွင့်ပြုချက်မရှိသော အသုံးပြုသူများသည် အခြားကဏ္ဍများ၏ dashboard များကို ဝင်ရောက်ကြည့်ရှု၍မရပါ။
- [ ] SQL injection နှင့် CSRF အားနည်းချက်များကို လျော့ပါးသက်သာစေသည်။
- [ ] File storage abstraction ကို အကောင်အထည်ဖော်ထားသည်; business logic ကို မပြောင်းလဲဘဲ implementation များကို ပြောင်းလဲနိုင်သည်။
- [ ] EF Core Migrations မှတစ်ဆင့် Database schema ကို ဖန်တီးပြီး ဗားရှင်းသတ်မှတ်ထားသည်။

### 7.3 အောင်မြင်မှု တိုင်းတာချက်များ (Success Metrics - Post-Implementation)

- **ပထမဆုံး အတည်ပြုချက်အတွက် ကြာချိန် (Time to First Approval):** တောင်းဆိုလွှာ တင်သွင်းချိန်မှ မန်နေဂျာ၏ ပထမဆုံး အတည်ပြုချက်အထိ ပျမ်းမျှအချိန် (ပစ်မှတ်: < ၂၄ နာရီ)။
- **အစမှအဆုံး လုပ်ငန်းစဉ် ကြာချိန် (End-to-End Cycle Time):** တောင်းဆိုလွှာ ဖန်တီးချိန်မှ ငွေပေးချေမှု ပြီးစီးချိန်အထိ ပျမ်းမျှအချိန် (ပစ်မှတ်: < ၅ ရက် - business days)။
- **စနစ် အလုပ်လုပ်ချိန် (System Uptime):** > 99% ရရှိနိုင်မှု (ကြိုတင်သတ်မှတ်ထားသော ပြုပြင်ထိန်းသိမ်းမှု ဖယ်ထုတ်ပြီး)။
- **အသုံးပြုသူ လက်ခံသုံးစွဲမှု (User Adoption):** ရည်ရွယ်ထားသော အသုံးပြုသူများ၏ > 90% သည် ၃ လအတွင်း စနစ်ကို တက်ကြွစွာ အသုံးပြုခြင်း။

## 8A. နောက်ဆက်တွဲ- ငွေပေးချေမှု တောင်းဆိုလွှာ ဖောင်ပုံစံ ရည်ညွှန်းချက် (Appendix: Payment Request Form Template Reference)

### ဖောင်ပုံစံ- 支払申請書 (Payment Request Form)

**ဖိုင်အမည်:** `支払申請書_YYYYMMDD_社員番号_社員名.xlsx`

ဤစနစ် လိုအပ်ချက်များ သတ်မှတ်ချက်သည် အဖွဲ့အစည်းတွင် အသုံးပြုသော စံသတ်မှတ်ထားသည့် ငွေပေးချေမှု တောင်းဆိုလွှာ ဖောင်ပုံစံ (支払申請書) ကို အခြေခံထားသည်။ ဖောင်ပုံစံ တည်ဆောက်ပုံကို စနစ် entity များနှင့် လုပ်ဆောင်ချက်ဆိုင်ရာ လိုအပ်ချက်များအဖြစ် အောက်ပါအတိုင်း ပြောင်းလဲထားသည်-

#### Template တည်ဆောက်ပုံ ပုံဖော်ခြင်း (Template Structure Mapping)

| ဖောင် အပိုင်း (Form Section) | စနစ် Entity / အပိုင်း (System Entity / Section) | ညွှန်းဆိုမှု (Mapped To) |
|---|---|---|
| Form Header (申請日, 社員番号, 氏名) | User & Request Header | Section 3.1.2, 3.5.1 |
| Payment Details (支払金額, 支払希望日, 通貨選択, စသည်) | PaymentRequest Entity | Section 3.1.2, 3.5.2 |
| Request Content (支払申請内容, 領収書の有無) | PaymentRequest Entity | Section 3.1.2, 3.5.3 |
| Payment Breakdown Table (支払内訳) | PaymentBreakdownItem Entity | Section 3.1.3, 3.5.4 |
| Receipt Attachment Area | Receipt File Storage | Section 3.4.6, 3.5.5 |

#### ဖောင် အကွက်များ ဘာသာပြန် ရည်ညွှန်းချက် (Form Fields Translation Reference)

Excel template မှ အောက်ပါ အကွက်များကို စနစ် လိုအပ်ချက်များတွင် ထည့်သွင်းထားသည်-

- **支払申請書** → Payment Request Form (စနစ် ခေါင်းစဉ်)
- **申請日** → Application Date (မဖြစ်မနေ လိုအပ်သည်, REQ-002A)
- **社員番号** → Employee Number (User profile မှ အလိုအလျောက် ဖြည့်ပေးသည်, REQ-002A)
- **氏名** → Employee Name (User profile မှ အလိုအလျောက် ဖြည့်ပေးသည်, REQ-002A)
- **下記の通り支払い申請を致します** → ငွေပေးချေမှု တောင်းဆိုခြင်းကို ညွှန်ပြသော ဖော်ပြချက် (ဖောင် နိဒါန်း စာသား)
- **支払金額** → Total Payment Amount (အလိုအလျောက် တွက်ချက်သည်, REQ-002A)
- **支払希望日** → Desired Payment Date (မဖြစ်မနေ လိုအပ်သည်, REQ-002A)
- **支払申請内容** → Payment Request Content (မဖြစ်မနေ လိုအပ်သည်, REQ-002A)
- **支払タイプ** → Payment Type (မဖြစ်မနေရွေးချယ်ရမည့် dropdown, REQ-002A)
- **支払方法** → Payment Method (မဖြစ်မနေရွေးချယ်ရမည့် dropdown, REQ-002A)
- **領収書の有無** → Receipt Present (Yes/No, REQ-002A)
- **用途** → Purpose/Usage (မဖြစ်မနေ လိုအပ်သည်, REQ-002A)
- **通貨選択** → Currency Type (မဖြစ်မနေရွေးချယ်ရမည့် dropdown, REQ-002A)
- **銀行口座・口座名/電話番号** → Bank Account / Account Name / Phone (အခြေအနေအရ လိုအပ်သည်, REQ-002A)
- **支払内訳** → Payment Breakdown Table (မဖြစ်မနေ လိုအပ်သည်, REQ-002B)
  - **No** → Line Number (1-15, အလိုအလျောက် နံပါတ်စဉ်တပ်သည်)
  - **日付** → အသုံးစရိတ် ရက်စွဲ
  - **内容** → breakdown item ၏ ဖော်ပြချက်
  - **合計** → Total (တွက်ချက်ထားသော စုစုပေါင်း)

#### အရေးကြီးသော ဖောင် မှတ်စုများ (Important Form Notes - From Template)

ဖောင်တွင် အောက်ပါ အရေးကြီးသော မှတ်စုများ ပါဝင်ပြီး ၎င်းတို့ကို လုပ်ငန်းစည်းမျဉ်းများ (business rules) အဖြစ် ထည့်သွင်းထားသည်-

1. **ပြေစာ လိုအပ်ချက် မှတ်စု (Receipt Requirement Note):**
   - ဂျပန်စာ: "※支払い申請時には、基本的に領収書の提出が必要です。"
   - အင်္ဂလိပ်: "Receipt file submission is basically required when applying for payment."
   - အကောင်အထည်ဖော်မှု (Implementation): REQ-029A, စည်းမျဉ်း 4.1.6

2. **ပြေစာဖိုင် အမည်ပေးခြင်း စည်းမျဉ်း မှတ်စု (Receipt File Naming Convention Note):**
   - ဂျပန်စာ: "領収書のファイル名は「支払内訳」の内容に従って設定してください。"
   - အင်္ဂလိပ်: "Receipt file names should follow the Payment Breakdown content format."
   - အကောင်အထည်ဖော်မှု (Implementation): REQ-030A, စည်းမျဉ်း 4.2.3

3. **စက္ကူပြေစာ တင်သွင်းခြင်း မှတ်စု (Paper Receipt Submission Note):**
   - ဂျပန်စာ: "※紙の領収書は、支出後または申請後に経理へ提出してください。"
   - အင်္ဂလိပ်: "Paper receipts should be submitted to Accounting after payment or after request submission."
   - အကောင်အထည်ဖော်မှု (Implementation): REQ-032A, စည်းမျဉ်း 4.1.7

## အတည်ပြု လက်မှတ်ရေးထိုးခြင်း (Sign-Off)

ဤ လိုအပ်ချက်များ သတ်မှတ်ချက် စာရွက်စာတမ်းသည် ငွေပေးချေမှု တောင်းဆိုခြင်း လုပ်ငန်းအဆင့်ဆင့် စီမံခန့်ခွဲမှုစနစ် (Payment Request Workflow Management System) အတွက် သဘောတူညီထားသော သတ်မှတ်ချက်ကို ကိုယ်စားပြုသည်။ ဖွံ့ဖြိုးတိုးတက်မှု (development) မစတင်မီ သက်ဆိုင်သူအားလုံးမှ ပြန်လည်သုံးသပ်ပြီး အတည်ပြုရမည်။

**အတည်ပြုချက် အခြေအနေ:** ပြန်လည်သုံးသပ်ရန် စောင့်ဆိုင်းနေသည် (Pending Review)  
**နောက်တစ်ဆင့်:** နည်းပညာပိုင်းဆိုင်ရာ ဒီဇိုင်း (Technical Design - 技術設計) နှင့် Database Schema ဒီဇိုင်း
