## 1. လုပ်ဆောင်ချက် အကျဉ်းချုပ်

### 1.1 ရည်ရွယ်ချက်နှင့် နယ်ပယ်

ဤမျက်နှာပြင်သည် ငွေပေးချေမှု တောင်းခံလွှာ လုပ်ငန်းစဉ် စီမံခန့်ခွဲမှု စနစ်အတွင်း `APPROVER` (နောက်ဆုံး အတည်ပြုသူ) အခန်းကဏ္ဍ သတ်မှတ်ခံရသော အသုံးပြုသူများအတွက် အဓိက လည်ပတ်မှု ပေါ်တယ် (portal) ဖြစ်သည်။ ၎င်းသည် ငွေပေးချေမှု တောင်းခံလွှာ ဘဝစက်ဝန်း၏ နောက်ဆုံး ဆုံးဖြတ်ချက်ချသည့် အဆင့်ကို စီမံခန့်ခွဲသည်။ ဤမျက်နှာပြင်သည် နောက်ဆုံး အတည်ပြုသူအား မန်နေဂျာမှ စစ်ဆေးအတည်ပြုပြီးသော အချက်အလက်များကို စစ်ဆေးရန်၊ ငွေပေးချေမှု အသေးစိတ်နှင့် ပြေစာ ဖိုင်များကို ကြည့်ရှုရန်၊ ပြီးပြည့်စုံသော အတည်ပြုချက်မှတ်တမ်း (`ApprovalLog`) ကို ပြန်လည်သုံးသပ်ရန်နှင့် တောင်းခံလွှာအား အတည်ပြုရန် သို့မဟုတ် ပယ်ချရန်အတွက် ဆုံးဖြတ်ချက်ချနိုင်စေရန် လုပ်ဆောင်ပေးသည်။

ဤလုပ်ဆောင်ချက်သည် ဒုတိယအဆင့် စစ်ဆေးအတည်ပြုချက် တံခါးပေါက်ဖြစ်ပြီး ၎င်းပြီးဆုံးပါက ငွေစာရင်းဌာန (Accounting queue) သို့ ရောက်ရှိမည် ဖြစ်သည်။

### 1.2 လုပ်ဆောင်ချက်ဆိုင်ရာ တာဝန်များ

ဤမျက်နှာပြင်သည် အောက်ပါ အဓိက လုပ်ဆောင်ချက်ဆိုင်ရာ နယ်ပယ်များအတွက် တာဝန်ရှိသည်-

1. **ဆိုင်းငံ့အတည်ပြုချက်များ စာရင်းအား ပြသခြင်း (Pending Approvals Queue Display)** — နောက်ဆုံး အတည်ပြုသူ၏ အရေးယူဆောင်ရွက်မှုကို စောင့်ဆိုင်းနေသော အခြေအနေများ (`Submitted to Approver` သို့မဟုတ် `Approver Reviewing`) တွင်ရှိသည့် ငွေပေးချေမှု တောင်းခံလွှာများအားလုံးကို စာရင်းအဖြစ် တင်ဆက်ခြင်း။
2. **ဖွင့်လှစ်ချိန်တွင် အခြေအနေ အလိုအလျောက် ပြောင်းလဲခြင်း (Automatic Status Transition on Open)** — `Submitted to Approver` အခြေအနေရှိသော တောင်းခံလွှာတစ်ခုကို နောက်ဆုံး အတည်ပြုသူမှ ဖွင့်လှစ်လိုက်သည့်အခါ အခြားအသုံးပြုသူများ အချိန်နှင့်တစ်ပြေးညီ သိရှိနိုင်ရန်အတွက် တောင်းခံလွှာ၏ အခြေအနေအား `Approver Reviewing` သို့ စနစ်မှ အလိုအလျောက် အသစ်ပြင်ဆင်ခြင်း။
3. **ဖတ်ရန်သာဖြစ်သော အသေးစိတ် အချက်အလက်များအား ပြသခြင်း (Comprehensive Read-Only Review)** — ခေါင်းစဉ် အချက်အလက်များ၊ ငွေပေးချေမှု ပမာဏများ၊ လိုင်းအသေးစိတ် အချက်အလက်များနှင့် ပူးတွဲထားသော ပြေစာဖိုင်များကို ဖတ်ရန်သာ (read-only) ဗားရှင်းဖြင့် ပြသပေးခြင်း။
4. **အတည်ပြုချက်မှတ်တမ်းအား ပြန်လည်သုံးသပ်ခြင်း (Approval Log Audit Trail Review)** — ယခင်က ပြုလုပ်ခဲ့သော အတည်ပြုမှု လုပ်ဆောင်ချက်များ၊ တာဝန်ရှိသူအမည်များ၊ အခြေအနေ ပြောင်းလဲခဲ့သည့် ရက်စွဲများနှင့် မှတ်ချက်များကို အချိန်အစီအစဉ်အလိုက် ဖော်ပြပေးခြင်း။
5. **အတည်ပြုခြင်း လုပ်ဆောင်ချက် (Approval Action)** — တောင်းခံလွှာ အခြေအနေအား `Approved` သို့ ပြောင်းလဲခြင်း၊ တာဝန်ပေးအပ်မှုကို ငွေစာရင်းဌာန (Accounting) သို့ လွှဲပြောင်းခြင်းနှင့် ပြင်ဆင်၍မရသော အတည်ပြုချက်မှတ်တမ်းထဲတွင် မှတ်တမ်းတင်ခြင်း။
6. **ပယ်ချခြင်း လုပ်ဆောင်ချက် (Rejection Action)** — အနည်းဆုံး စာလုံးရေ ၁၀ လုံးပါဝင်သော မဖြစ်မနေ လိုအပ်သည့် မှတ်ချက်ကို အတည်ပြုစစ်ဆေးခြင်း၊ တောင်းခံလွှာ အခြေအနေအား `Rejected by Approver` သို့ ပြောင်းလဲခြင်း၊ တာဝန်ပေးအပ်မှုကို လျှောက်ထားသူထံ ပြန်လည်ပေးအပ်ခြင်းနှင့် အတည်ပြုချက်မှတ်တမ်းထဲတွင် မှတ်တမ်းတင်ခြင်း။

### 3.3 တည်းဖြတ်နိုင်သော အခြေအနေ အစုအဝေး သတ်မှတ်ချက်

ဤသရုပ်ဆောင်မှ ရေးသားနိုင်သော တစ်ခုတည်းသော အကွက်မှာ ပယ်ချသည့် modal မျက်နှာပြင်ရှိ `comment` ထည့်သွင်းသည့် အကွက်နှင့် အတည်ပြု/ပယ်ချမှု လုပ်ဆောင်ချက် ခလုတ်များသာ ဖြစ်သည်။

### 5.2 မျက်နှာပြင်: တောင်းခံလွှာ အသေးစိတ် မြင်ကွင်း (`/approver/requests/:id` သို့မဟုတ် Modal)

**ရည်ရွယ်ချက်:** တောင်းခံလွှာတစ်ခုတည်း၏ အချက်အလက်အပြည့်အစုံ၊ သမိုင်းကြောင်းနှင့် အတည်ပြုချက် လုပ်ဆောင်ချက်များကို ပြသရန်။

#### 5.2.1 ဖတ်ရန်သာဖြစ်သော အသေးစိတ် အပိုင်းများ

**လျှောက်ထားသူ အချက်အလက်:**
* 社員番号 / ဝန်ထမ်း နံပါတ် (`employee_number`)
* 氏名 / အမည်အပြည့်အစုံ (`full_name`)
* 支店名 / ဌာနခွဲ (`branch`)
* 部署名 / ဌာန (`department`)

**ငွေပေးချေမှု အချက်အလက် ခေါင်းစဉ်:**
* 申請日 / လျှောက်ထားသည့် ရက်စွဲ (`application_date`) — YYYY-MM-DD
* 支払希望日 / မျှော်မှန်း ငွေပေးချေရက် (`desired_payment_date`) — YYYY-MM-DD
* 支払金額 / စုစုပေါင်း ပမာဏ (`total_amount`) — e.g. USD 1,500.00
* 通貨選択 / Ngwe kyay aamyoasarr — e.g. USD
* 支払タイプ / Ngwepeychaymhu Aamyoasarr — e.g. Expense Reimbursement
* 支払方法 / Ngwepeychaymhu Nannlamm — e.g. Bank Transfer
* 銀行口座情報 / ဘဏ်အကောင့် အချက်အလက် (`bank_account_info`) — (နည်းလမ်းမှာ Bank Transfer သို့မဟုတ် Cash ဖြစ်ပါကသာ ပြသမည်)
* 用途 / ရည်ရွယ်ချက် (`purpose`)

**တောင်းဆိုချက် အသေးစိတ်နှင့် ပူးတွဲဖိုင်များ:**
* 支払申請内容 / အသေးစိတ် အကြောင်းအရာ ဖော်ပြချက် (`request_content`)
* 領収書の有無 / ပြေစာ ပူးတွဲမှု အခြေအနေ (`has_receipt`) — ရှိသည် / မရှိပါ
* 添付ファイル / တင်ထားသော ပြေစာဖိုင်များ — ဒေါင်းလုဒ်ဆွဲရန် လင့်ခ်များစာရင်း: ဥပမာ- `[OfficeSupplies_20260609_01.pdf]`

**支払内訳 / ငွေပေးချေမှု အသေးစိတ်လိုင်း ဇယား:**
* `လိုင်းနံပါတ်`၊ `ရက်စွဲ`၊ `ဖော်ပြချက်`၊ `ပမာဏ` ကော်လံများ ပါဝင်သော ဇယား (ပေါင်းလဒ်သည် စုစုပေါင်း ပမာဏနှင့် ကိုက်ညီရမည်)။

**承認履歴 / အတည်ပြုချက် သမိုင်းကြောင်း အချိန်ဇယား:**
* `approval_logs` ဒေတာကို အချိန်အစီအစဉ်အလိုက် ပြသသောစာရင်း။ ဖော်ပြမည့်အချက်အလက်များ-
  * နေ့ရက်နှင့် အချိန် (အသုံးပြုသူ၏ ဒေသစံတော်ချိန်သို့ ပြောင်းလဲထားသည်)
  * လုပ်ဆောင်သူ အမည်နှင့် အခန်းကဏ္ဍ
  * လုပ်ဆောင်ချက် အညွှန်း (ဥပမာ- Created, Submitted, Manager Verified, Approver Review Started)
  * ပယ်ချသည့် မှတ်ချက်များ (ရှိပါက italic စာလုံးစောင်း blockquote ဖြင့် ပြသမည်)

## 6. လုပ်ဆောင်ချက် လည်ပတ်မှုဆိုင်ရာ သတ်မှတ်ချက်

### 6.1 လုပ်ဆောင်ချက်: ဆိုင်းငံ့စာရင်းကို ရယူခြင်း (Fetch Pending Queue List)

| အရည်အချင်း | သတ်မှတ်ချက် |
| :--- | :--- |
| **အစပျိုးမှု** | ဒက်ရှ်ဘုတ် စတင်ပွင့်လာချိန်၊ စာမျက်နှာပြောင်းလဲချိန် သို့မဟုတ် refresh ခလုတ်ကို နှိပ်လိုက်ချိန်။ |
| **ဒေတာ ရင်းမြစ်** | API အမှတ်အသား: `GET /api/payment-requests/approver/pending` |
| **စုံစမ်းမှု စစ်ထုတ်ချက်** | `status_id IN (SUBMITTED_APPROVER, APPROVER_REVIEWING) AND is_deleted = FALSE` |
| **ပုံသေ စီစဉ်မှု** | `application_date ASC` |
| **စာမျက်နှာခွဲခြားမှု** | Server ဘက်မှ စာမျက်နှာ ခွဲခြားမှု။ ပုံသေ- စာမျက်နှာတစ်ခုလျှင် မှတ်တမ်း ၂၀။ |
| **ပြသမည့် အကွက်များ** | တောင်းခံလွှာ နံပါတ်၊ လျှောက်ထားသူ အမည်၊ ဌာနခွဲ၊ လျှောက်ထားသည့်ရက်၊ စုစုပေါင်းပမာဏ၊ မျှော်မှန်းငွေပေးချေရက်၊ အခြေအနေ အမည်။ |
| **အချိန်နှင့်တစ်ပြေးညီ မွမ်းမံမှု** | အခြေအနေ ပြောင်းလဲမှုအတွက် `statusUpdate` WebSocket ပွဲစဉ်ကို လက်ခံရရှိပါက ဇယားအား အလိုအလျောက် အသစ်ပြန်လည်ရယူသည်။ |

### 6.2 လုပ်ဆောင်ချက်: တောင်းခံလွှာ အသေးစိတ်အား ဖွင့်လှစ်ခြင်း (ဖွင့်ချိန်တွင် အခြေအနေ အလိုအလျောက် ပြောင်းလဲခြင်း)

| အရည်အချင်း | သတ်မှတ်ချက် |
| :--- | :--- |
| **အစပျိုးမှု** | အသုံးပြုသူသည် ဇယားရှိ တောင်းခံလွှာနံပါတ် လင့်ကို နှိပ်လိုက်ချိန်။ |
| **အစောင့် စည်းမျဉ်း** | JWT အခန်းကဏ္ဍ စစ်ဆေးမှု (`role_code = 'APPROVER'`)။ တောင်းခံလွှာ တည်ရှိပြီး အသက်ဝင်နေရမည်။ |
| **လုပ်ဆောင်မှု အဆင့်များ**| 1. တောင်းခံလွှာ အသေးစိတ်၊ အသေးစိတ်လိုင်းများ၊ ပြေစာလမ်းကြောင်းများနှင့် အတည်ပြုချက်မှတ်တမ်းများကို မေးမြန်းရယူသည်။<br>2. **အခြေအနေသည် `SUBMITTED_APPROVER` ဖြစ်နေပါက:**<br>&nbsp;&nbsp;&nbsp;&nbsp;a. Database Transaction ကို စတင်သည်။<br>&nbsp;&nbsp;&nbsp;&nbsp;b. `payment_requests` ဇယားတွင် `status_id = APPROVER_REVIEWING` နှင့် `modified_date = CURRENT_TIMESTAMP` သို့ ပြင်ဆင်သည်။ `current_assigned_to_user_id` အား လက်ရှိ Approver အသုံးပြုသူအဖြစ် သတ်မှတ်သည်။<br>&nbsp;&nbsp;&nbsp;&nbsp;c. `approval_logs` ထဲသို့ မှတ်တမ်းအသစ်တစ်ခု ထည့်သွင်းသည် (`action_type_id` အား `APPR_REVIEW_START` သို့ သတ်မှတ်ပြီး၊ ယခင်အခြေအနေ `SUBMITTED_APPROVER`၊ အခြေအနေအသစ် `APPROVER_REVIEWING` ဟု သတ်မှတ်သည်)။<br>&nbsp;&nbsp;&nbsp;&nbsp;d. Transaction ကို Commit ပြုလုပ်သည်။<br>&nbsp;&nbsp;&nbsp;&nbsp;e. Redis cache ဒေတာကို ဖျက်ပစ်သည် (`DEL payment_request:payload:<id>`)။<br>&nbsp;&nbsp;&nbsp;&nbsp;f. အခြေအနေအား Reviewing သို့ ပြောင်းလဲသွားကြောင်း အသိပေးရန် WebSocket `statusUpdate` ပွဲစဉ်ကို ပေးပို့သည်။<br>3. ရရှိလာသော ဒေတာများဖြင့် အသေးစိတ် မြင်ကွင်း UI ကို တင်ဆက်ပြသသည်။ |
| **တစ်ပြိုင်တည်း လုပ်ဆောင်မှု ထိန်းချုပ်ခြင်း**| အခြေအနေအပေါ် 楽観的ロック (Optimistic lock) နှိုင်းယှဉ်မှုကို ပြုလုပ်သည်။ Transaction မလုပ်ဆောင်မီ အခြေအနေသည် `SUBMITTED_APPROVER` မှ ပြောင်းလဲသွားပါက ပဋိပက္ခအမှား (HTTP 409) ကို ပြန်လည်ပေးပို့သည်။ |

### 6.3 လုပ်ဆောင်ချက်: အတည်ပြုခြင်းအား အတည်ပြုချက်ပေးခြင်း (Confirm Approval)

| အရည်အချင်း | သတ်မှတ်ချက် |
| :--- | :--- |
| **အစပျိုးမှု** | အတည်ပြုသူသည် "Approve" ကို နှိပ်ပြီး တွေ့ဆုံဆွေးနွေးမှု modal တွင် အတည်ပြုလိုက်ချိန်။ |
| **အစောင့် စည်းမျဉ်း** | တောင်းခံလွှာ အခြေအနေသည် `APPROVER_REVIEWING` ဖြစ်ရမည်။ အထောက်အထား စစ်ဆေးပြီးသား အသုံးပြုသူသည် `APPROVER` ဖြစ်ရမည်။ |
| **API အမှတ်အသား** | `POST /api/payment-requests/:id/approve` |
| **ထည့်သွင်းမှု ဒေတာ** | `{}` (လမ်းကြောင်း ID မှတစ်ဆင့် ကန့်သတ်ချက်များကို ရယူသည်) |
| **လုပ်ဆောင်မှု အဆင့်များ**| 1. Database Transaction ကို စတင်သည်။<br>2. တောင်းခံလွှာ အခြေအနေသည် `APPROVER_REVIEWING` ဖြစ်ကြောင်း စစ်ဆေးသည်။<br>3. `payment_requests` ဇယားတွင် `status_id = APPROVED`၊ `approval_date = CURRENT_TIMESTAMP`၊ `modified_date = CURRENT_TIMESTAMP`၊ `final_approver_user_id = current_user.id` နှင့် `current_assigned_to_user_id = NULL` (Accounting queue သို့ လမ်းကြောင်းညွှန်ပြသည်) သို့ အသစ်ပြင်ဆင်သည်။<br>4. `approval_logs` ထဲသို့ `APPROVED` လုပ်ဆောင်ချက်၊ ယခင်အခြေအနေ `APPROVER_REVIEWING`၊ အခြေအနေအသစ် `APPROVED` တို့ဖြင့် မှတ်တမ်းထည့်သွင်းသည်။<br>5. Transaction ကို Commit ပြုလုပ်သည်။<br>6. Redis cache ဒေတာကို ဖျက်ပစ်သည် (`DEL payment_request:payload:<id>`)။<br>7. WebSocket `statusUpdate` ပွဲစဉ်ကို လွှင့်ထုတ်သည် (status အား `APPROVED` သို့ သတ်မှတ်သည်)။ |
| **တုံ့ပြန်မှု ဒေတာ** | `{ "success": true, "message": "Request successfully approved." }` |

### 6.4 လုပ်ဆောင်ချက်: ပယ်ချခြင်းအား အတည်ပြုချက်ပေးခြင်း (Confirm Rejection)

| အရည်အချင်း | သတ်မှတ်ချက် |
| :--- | :--- |
| **အစပျိုးမှု** | အတည်ပြုသူသည် ပယ်ချသည့် မှတ်ချက်ကို ရေးသားပြီး modal တွင် "Submit Rejection" ခလုတ်ကို နှိပ်လိုက်ချိန်။ |
| **အစောင့် စည်းမျဉ်း** | တောင်းခံလွှာ အခြေအနေသည် `APPROVER_REVIEWING` ဖြစ်ရမည်။ ပယ်ချသည့် မှတ်ချက်သည် အတည်ပြုစစ်ဆေးမှု အောင်မြင်ရမည် (မပြည့်မစုံ ဖြစ်နေခြင်း မရှိရ၊ အနည်းဆုံး စာလုံးရေ ၁၀ လုံး ရှိရမည်)။ |
| **API အမှတ်အသား** | `POST /api/payment-requests/:id/reject-approver` |
| **ထည့်သွင်းမှု ဒေတာ** | `{ "comment": "String (minimum 10 characters)" }` |
| **လုပ်ဆောင်မှု အဆင့်များ**| 1. မှတ်ချက် စာလုံးရေကို စစ်ဆေးသည်။ မအောင်မြင်ပါက လုပ်ဆောင်မှုကို ဖျက်သိမ်းပြီး validation logic အမှားကို ပြန်ပေးသည်။<br>2. Database Transaction ကို စတင်သည်။<br>3. တောင်းခံလွှာ အခြေအနေသည် `APPROVER_REVIEWING` ဖြစ်ကြောင်း စစ်ဆေးသည်။<br>4. လျှောက်ထားသူ ID (`applicant_user_id`) ကို ရယူသည်။<br>5. `payment_requests` ဇယားတွင် `status_id = REJECTED_APPROVER`၊ `modified_date = CURRENT_TIMESTAMP`၊ `final_approver_user_id = current_user.id` နှင့် `current_assigned_to_user_id = applicant_user_id` (တာဝန်ပေးအပ်မှုကို လျှောက်ထားသူထံ ပြန်ပေးသည်) သို့ အသစ်ပြင်ဆင်သည်။<br>6. `approval_logs` ထဲသို့ `APPR_REJECTED` လုပ်ဆောင်ချက်၊ ယခင်အခြေအနေ `APPROVER_REVIEWING`၊ အခြေအနေအသစ် `REJECTED_APPROVER` နှင့် အသုံးပြုသူ၏ မှတ်ချက်တို့ကို ထည့်သွင်းမှတ်တမ်းတင်သည်။<br>7. Transaction ကို Commit ပြုလုပ်သည်။<br>8. Redis cache ဒေတာကို ဖျက်ပစ်သည် (`DEL payment_request:payload:<id>`)။<br>9. WebSocket `statusUpdate` ပွဲစဉ်ကို လွှင့်ထုတ်သည် (status အား `REJECTED_APPROVER` သို့ သတ်မှတ်သည်)။ |
| **တုံ့ပြန်မှု ဒေတာ** | `{ "success": true, "message": "Request successfully rejected and returned to applicant." }` |

## 7. ထည့်သွင်းမှု / ထွက်ပေါ်လာမှု သတ်မှတ်ချက်

### 7.1 ထည့်သွင်းမှု သတ်မှတ်ချက် (入力定義)

ဤမျက်နှာပြင်သည် တောင်းခံလွှာ တန်ဖိုးများအတွက် ဖတ်ရန်သာ ဖြစ်သည်။ အပြန်အလှန် တုံ့ပြန်နိုင်သော တစ်ခုတည်းသော ထည့်သွင်းမှုများမှာ-

| အကွက်အမည် (Physical Name) | မျက်နှာပြင် အမည် | ဒေတာ အမျိုးအစားနှင့် အရှည် | မဖြစ်မနေ လိုအပ်သည် | ထိန်းချုပ်မှု ပုံစံ | အတည်ပြုစစ်ဆေးခြင်း စည်းမျဉ်းများ / မှတ်ချက် |
| :--- | :--- | :--- | :---: | :--- | :--- |
| `comment` | ပယ်ချသည့် မှတ်ချက် | TEXT (စာလုံးရေ ၁၀၀၀ ထိ) | Yes (ပယ်ချသည့်အခါ၌သာ) | Rejection Modal ရှိ စာသားရေးရန်နေရာ | ပယ်ချသည့်အခါ မဖြစ်မနေ လိုအပ်သည်။ အနည်းဆုံး စာလုံးရေ ၁၀ လုံး ရှိရမည်ဟု ကန့်သတ်ထားသည်။ |

### 7.2 ထွက်ပေါ်လာမှု သတ်မှတ်ချက် (出力定義)

အသေးစိတ် မြင်ကွင်းတွင် ပြသထားသော အချက်အလက်များသည် **အပိုင်း 5.2.1** ရှိ သတ်မှတ်ချက်များနှင့် ကိုက်ညီပြီး အောက်ပါတို့မှ ဖတ်ရှုခြင်းဖြစ်သည်-
* `payment_requests` ဇယား
* `payment_breakdown_items` ဇယား
* `receipt_files` ဇယား
* `approval_logs` ဇယား

## 9. အမှားအယွင်း ကိုင်တွယ်ဖြေရှင်းခြင်း သတ်မှတ်ချက်

### 9.1 အမှားအယွင်း အမျိုးအစား ခွဲခြားမှု ဇယား

| HTTP အခြေအနေ | အမှားကုဒ် | ဖြစ်ပေါ်သည့် အခြေအနေ | အသုံးပြုသူ မြင်တွေ့ရမည့် အပြုအမူ |
| :---: | :--- | :--- | :--- |
| `400` | `BAD_REQUEST` | ပယ်ချမှုကို တင်ပြချိန်တွင် မှတ်ချက် မပါဝင်ခြင်း သို့မဟုတ် စာလုံးရေ ၁၀ လုံးအောက် နည်းနေခြင်း။ | မှတ်ချက်အကွက်အား အနီရောင်ဘောင်ဖြင့် ပြသပြီး အမှားစာသားကို ဘေးတွင် ဖော်ပြသည်။ |
| `403` | `FORBIDDEN` | အထောက်အထား စစ်ဆေးထားသော အသုံးပြုသူသည် Approver မဟုတ်ခြင်း သို့မဟုတ် အခြေအနေ `SUBMITTED_APPROVER` သို့မဟုတ် `APPROVER_REVIEWING` မဟုတ်သော တောင်းခံလွှာအသေးစိတ်ကို ဝင်ရောက်ရန် ကြိုးပမ်းခြင်း။ | ဝင်ရောက်မှုကို ပိတ်ပင်ပြီး "Access denied. You do not have the required permissions for this action." ဟု ပြသသည်။ |
| `409` | `CONFLICT` | အသုံးပြုသူမှ အသေးစိတ်မျက်နှာပြင်ကို ဖွင့်ထားစဉ် အခြားအတည်ပြုသူတစ်ဦးမှ တောင်းခံလွှာကို အတည်ပြု/ပယ်ချ လုပ်ဆောင်သွားခြင်း (樂観的ロック ပဋိပက္ခ)။ | Dialog ဖြင့် ပြသမည်- "This request has already been processed by another user. Dashboard has been updated." ပြီးနောက် ဒက်ရှ်ဘုတ်စာရင်းသို့ ပြန်လည်လမ်းညွှန်ပေးသည်။ |
| `500` | `INTERNAL_SERVER_ERROR` | Commit ပြုလုပ်စဉ် လိုင်းချိတ်ဆက်မှု ပြတ်တောက်ခြင်း သို့မဟုတ် database transaction ပျက်စီးခြင်း။ | Toast ဖြင့် ပြသမည်- "A system error occurred while saving approval details. Please try again." |

## 10. ခွင့်ပြုချက်နှင့် ဝင်ရောက်ခွင့် ထိန်းချုပ်မှု

### 10.1 အထောက်အထား စစ်ဆေးမှု လိုအပ်ချက်များ

စံနှုန်းသတ်မှတ်ထားသော JSON Web Token (JWT) အတည်ပြုခြင်းကို ပြဋ္ဌာန်းသည်။ Token ကို standard Bearer HTTP header မှတစ်ဆင့် ပို့ဆောင်သည်။ အသုံးပြုသူ၏ session သက်တမ်းကို client ဘက်မှ အလိုအလျောက် သက်တမ်းတိုးစေသည်။

### 10.2 ခွင့်ပြုချက် အစောင့်အရှောက် ဗိသုကာ (Authorization Guard Architecture)

နောက်ဆုံး အတည်ပြုသူ၏ လုပ်ဆောင်ချက် API တိုင်းသည် အောက်ပါတို့ကို ဖြတ်သန်းရမည်-
1. `JwtAuthGuard` — Token ၏ တရားဝင်မှုကို စစ်ဆေးပြီး အသုံးပြုသူအချက်အလက်ကို ရယူသည်။
2. `RolesGuard` — အသုံးပြုသူတွင် `APPROVER` အခန်းကဏ္ဍ ကုဒ် ရှိမရှိ စစ်ဆေးသည်။
3. `AssignmentGuard` — အတည်ပြု/ပယ်ချမှုများအတွက် တောင်းခံလွှာ၏ အခြေအနေသည် `APPROVER_REVIEWING` ဖြစ်ပြီး လက်ရှိ တာဝန်ပေးအပ်ထားသူ ဟုတ်မဟုတ် စစ်ဆေးသည်။

### 10.3 API Endpoint ခွင့်ပြုချက် မက်ထရစ်

| Endpoint | Method | လိုအပ်သော အခန်းကဏ္ဍ | ဖော်ပြချက် |
| :--- | :---: | :--- | :--- |
| `/api/payment-requests/approver/pending` | `GET` | `APPROVER` | ဆိုင်းငံ့စာရင်းကို ရယူရန်။ |
| `/api/payment-requests/:id` | `GET` | `APPROVER` | တောင်းခံလွှာ တစ်ခုတည်း၏ အသေးစိတ်ကို ရယူရန် (လက်ရှိ အခြေအနေ `SUBMITTED_APPROVER` ဖြစ်ပါက `APPROVER_REVIEWING` သို့ အလိုအလျောက် ပြောင်းလဲမှု စတင်သည်)။ |
| `/api/payment-requests/:id/approve` | `POST` | `APPROVER` | အတည်ပြုခြင်း လုပ်ငန်းစဉ် အပြောင်းအလဲကို လုပ်ဆောင်ရန်။ |
| `/api/payment-requests/:id/reject-approver` | `POST` | `APPROVER` | ပယ်ချခြင်း လုပ်ငန်းစဉ် အပြောင်းအလဲကို လုပ်ဆောင်ရန်။ |

## 11. အချိန်နှင့်တစ်ပြေးညီ အသိပေးချက် အပြုအမူ

### 11.1 WebSocket ဖွဲ့စည်းမှုပုံစံ

* **ရုံးခန်း အဖွဲ့ဝင်မှု (Room Membership):** နောက်ဆုံး အတည်ပြုသူများသည် `'APPROVER'` ဟု အမည်ရသော အခန်းကဏ္ဍအလိုက် အခန်း (room) ထဲသို့ ဝင်ရောက်သည်။
* **အချိန်သတ်မှတ်ချက် လိုအပ်ချက်:** အခြေအနေ ပြောင်းလဲမှု အသိပေးချက်များကို transaction commit ပြီးနောက် ≤ ၅၀၀ မီလီစက္ကန့် အတွင်း ပို့ဆောင်ရမည်။

### 11.2 ဤမျက်နှာပြင်မှ လက်ခံရရှိမည့် အသိပေးချက် ပွဲစဉ်များ

* **ပွဲစဉ် `statusUpdate`:** အခြေအနေ ပြောင်းလဲမှု ဖြစ်ပေါ်သည့်အခါ စနစ်မှ ပေးပို့သည်။
  * *UI တုံ့ပြန်မှု:* အကယ်၍ ထို ID အတွက် အသေးစိတ် modal ပွင့်နေပါက သတိပေးချက် ပြသပြီး လုပ်ဆောင်ချက်များကို ပိတ်ပင်သည်။ ဆိုင်းငံ့စာရင်းကို ပြန်လည်မွမ်းမံသည်။

### 11.3 ဤမျက်နှာပြင်၏ လုပ်ဆောင်ချက်များမှ လွှင့်ထုတ်မည့် ပွဲစဉ်များ

* **အတည်ပြုသည့် လုပ်ဆောင်ချက်:** `'ACCOUNTING'` နှင့် `'user:<applicant_id>'` အခန်းများသို့ `statusUpdate` ကို လွှင့်ထုတ်သည်။
* **ပယ်ချသည့် လုပ်ဆောင်ချက်:** `'user:<applicant_id>'` အခန်းသို့ `statusUpdate` ကို လွှင့်ထုတ်သည်။

## 12. မျက်နှာပြင် ပြောင်းလဲမှုဆိုင်ရာ သတ်မှတ်ချက်

### 12.1 လမ်းညွှန်မှု စည်းမျဉ်းများ

* **အဝင် လမ်းကြောင်း (Inbound Navigation):** စနစ်သို့ ဝင်ရောက်ခြင်း -> အသုံးပြုသူ အခန်းကဏ္ဍသည် `APPROVER` ဖြစ်ပါက `/approver/dashboard` သို့ အလိုအလျောက် လမ်းညွှန်ပေးသည်။
* **အတွင်းပိုင်း အခြေအနေ လမ်းညွှန်မှု:** တောင်းခံလွှာ လင့်ခ်ကို နှိပ်ခြင်းဖြင့် dashboard စာရင်းမျက်နှာပြင်ပေါ်တွင် အသေးစိတ်ပြသသည့် Modal ကို ဖွင့်လှစ်ပေးသည်။
* **အထွက် လမ်းကြောင်း:** စနစ်မှ ထွက်ခွာခြင်း၊ session သက်တမ်းကုန်ဆုံးခြင်း သို့မဟုတ် လုပ်ဆောင်ချက် ပြီးမြောက်ခြင်း (modal ကို ပိတ်ပြီး dashboard စာရင်းကို ပြန်လည်ရယူသည်)။

## 13. လုပ်ဆောင်ချက်မဟုတ်သော ထည့်သွင်းစဉ်းစားမှုများ

### 13.1 စွမ်းဆောင်ရည် လိုအပ်ချက်များ

* စာမျက်နှာ တင်ဆက်ချိန် (Page render time): ≤ ၂ စက္ကန့်။
* အသေးစိတ်မြင်ကွင်း တင်ခြင်းနှင့် အလိုအလျောက် အခြေအနေ ပြောင်းလဲခြင်း လုပ်ဆောင်ချိန်: ≤ ၁.၅ စက္ကန့်။
* လုပ်ဆောင်ချက်များအတွက် API တုံ့ပြန်ချိန်: ≤ ၁ စက္ကန့် (index `idx_payment_requests_status_id` ဖြင့် အကောင်းဆုံးဖြစ်အောင် ပြုလုပ်ထားသည်)။

### 13.2 Cache မဟာဗျူဟာ

* **ဒေတာ Cache ပြုလုပ်ခြင်း:** တက်ကြွနေသော အသေးစိတ် JSON ဒေတာများကို Redis တွင် `payment_request:payload:<id>` အမည်ဖြင့် ၁၀ မိနစ် TTL သတ်မှတ်၍ cache ပြုလုပ်ထားသည်။
* **Cache ဖျက်သိမ်းခြင်း:** အတည်ပြု သို့မဟုတ် ပယ်ချမှု commit ပြုလုပ်ပြီးသည်နှင့်တစ်ပြိုင်နက် `DEL` ကို အသုံးပြု၍ ချက်ချင်း ဖျက်ပစ်သည်။
